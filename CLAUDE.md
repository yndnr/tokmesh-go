# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## 项目概览

这是一个**规格说明驱动开发 (SDD, Specification-Driven Development)** 项目模板，采用文档优先的开发方法论，强调通过结构化文档和交互式 AI 协作来驱动软件开发全过程。

**核心理念**：
- 文档是实施的依据，代码是文档的实现
- 所有功能必须先有 specs/ 文档
- AI 作为"结对设计"伙伴，深度参与从想法到实现的全流程

## AI 协作指南

本项目已配置专门的 AI 协作规范，工作前**必须**阅读：

1. **AGENTS.md** - AI 角色模型与交互规范
   - 定义了 5 种 AI 角色：编排代理、需求代理、设计代理、实现代理、审阅代理
   - 规定全程使用中文进行交流和文档生成
   - 要求输出精简、状态透明、文档优先

2. **specs/governance/interactive-workflow.md** - 交互式规约工作流
   - 定义了想法捕捉、需求细化、技术设计、开发分解等工作流
   - 提供自然语言触发指令（如 `我有一个想法`、`sdd: design`）
   - 描述了 AI 在每个工作流中的执行逻辑

## 文档架构

项目采用分层文档结构，所有规格说明文档位于 `specs/` 目录：

```
specs/
├── governance/          # 项目治理文档
│   ├── charter.md      # 项目宪法：使命、愿景、目标、非目标
│   ├── principles.md   # 架构原则：安全、性能、可靠性等维度
│   ├── conventions.md  # 编码与命名约定（Go 语言规范）
│   ├── glossary.md     # 术语词汇表
│   └── interactive-workflow.md  # AI 交互式工作流定义
├── 0-captures/         # CP 系列 - 想法捕捉
├── 1-requirements/     # RQ 系列 - 需求文档（功能/非功能/基础设施需求）
├── 2-designs/          # DN 系列 - 技术设计文档
├── 3-tasks/            # TK 系列 - 开发任务分解
└── adrs/               # AD 系列 - 架构决策记录
```

### 文档编号规则

所有文档采用统一编号格式：**{TYPE}-{yyWW}{seq}-{标题}.md**

- **TYPE**：文档类型（2 字母，如 CP/RQ/DN/TK/AD）
- **yy**：年份后两位（如 2025 → 25）
- **WW**：ISO 周数（01-53，周一为一周第一天）
- **seq**：两位数顺序号（01-99）
- **标题**：文档核心内容抽取

**文档系列**：
- **CP-{yyWW}{seq}-{标题}.md**: 想法捕捉（Capture），例如 CP-254901-核心定位架构目标.md
- **RQ-{yyWW}{seq}-{标题}.md**: 需求文档（Requirements），例如 RQ-254901-会话令牌存储引擎.md
- **DN-{yyWW}{seq}-{标题}.md**: 设计文档（Design），例如 DN-254901-存储引擎架构设计.md
- **TK-{yyWW}{seq}-{标题}.md**: 任务文档（Tasks），例如 TK-254901-实现内存存储引擎.md
- **AD-{yyWW}{seq}-{标题}.md**: 架构决策记录（Architecture Decision），例如 AD-254901-选择单写多读模型.md

## 工作流触发指令

在与 AI 协作时，可使用以下自然语言指令触发特定工作流：

| 指令 | 功能 | AI 行为 |
|------|------|---------|
| `我有一个想法` | 想法捕捉 | 交互式挖掘想法细节，生成 CP 文档 |
| `我想细化一个需求` | 需求细化 | 定位并细化 R 系列文档，提供直接修改或拆分方案 |
| `我们来设计一个功能` | 技术设计 | 基于 R 文档创建/更新 D 系列，引导架构、API、数据模型讨论 |
| `我想继续XXX的实现工作` | 开发分解 | 检查 D 文档完备性，生成 TK 任务列表 |
| `sdd: status` | 状态报告 | 生成项目规约状态总览 |
| `sdd: export [目标]` | 导出配置 | 导出 SDD 规范或项目初始化提示词 |

## 核心工作原则

### 1. 文档驱动开发流程

```
想法 (CP) → 需求 (R) → 设计 (D) → 任务 (TK) → 实现 (代码)
```

- 每个阶段都有对应的文档模板和评审标准
- 不允许跳过文档直接编码
- 设计文档必须处于"已批准"状态才能分解任务

### 2. AI 角色切换

根据当前工作阶段，AI 应明确声明当前角色：
- 讨论需求时 → 需求代理
- 进行设计时 → 设计代理
- 分解任务时 → 实现代理
- 审查文档时 → 审阅代理
- 协调多步骤工作时 → 编排代理

### 3. 交互式协作模式

- 采用 2-4 轮对话挖掘需求细节
- 提供多个方案供用户选择（如直接修改 vs 拆分子需求）
- 在执行变更前必须征得用户确认
- 发现设计不完整时，先报告缺失部分，再询问是否补全

### 4. 质量保证要求

- 测试覆盖率要求 ≥ 80%
- 测试不达标不允许合并
- 遵循 Go 语言官方编码规范（详见 specs/governance/conventions.md）
- 所有架构决策通过 ADR 记录

## 项目状态

当前为**项目模板阶段**，主要内容为：
- 完整的文档模板结构
- 定义完备的治理规范和工作流
- 尚无实际的需求、设计或代码实现

启动新项目时，应：
1. 填充 `specs/governance/charter.md` 定义项目使命和目标
2. 根据领域定制 `specs/governance/glossary.md` 术语表
3. 使用 `我有一个想法` 开始捕捉第一个功能需求

## 语言与格式要求

- **沟通语言**：全程使用中文（包括文档生成和对话）
- **引用路径**：使用项目相对路径（如 `specs/1-requirements/R001-xxx.md`）
- **输出格式**：精简要点，避免冗长铺陈；必要时使用代码块或表格
- **状态透明**：明确说明假设、风险、缺口和无法执行的操作

## 关键约束

1. **不得擅自删除或重写用户未明确授权的内容**
2. **避免破坏性命令**（如 `git reset --hard`、`rm -rf` 等）
3. **安全优先**：不在日志或示例中泄露敏感信息
4. **文档完整性**：设计文档中不允许存在 `[待定]`、`[空]` 等占位符即开始实现
5. **持续学习**：每个工作流结束后，AI 应反思并提出优化建议（需用户批准后才可更新规范）
