# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## 项目概览

TokMesh 是一套**面向身份认证（IAM/SSO）场景的专用 Session/Token 存储与管理系统**，目标是替代 Redis 在 SSO 中的会话存储角色。

**核心定位**：
- 这不是通用 KV 数据库，而是专门为身份认证流量优化的领域专用系统
- 原生支持多维会话索引（User/Client/App/Tenant/Device）
- 内置会话全生命周期管理（创建/校验/续期/撤销/TTL 清理）
- 安全优先架构（PKI/mTLS、零信任网络假设、双端口安全域隔离）

**当前状态**：
- 项目处于**极早期阶段**（P1 基础搭建），代码量约 235 行
- 已有**完整的设计文档体系**（SDD），位于 `docs/sdd/`
- 采用**文档驱动开发**（Doc-First）模式，所有代码必须先有设计文档支撑

## 代码架构

### 包结构（Package Structure）

```
cmd/
  tokmesh-server/          # 主服务入口（保持逻辑精简）
  tokmesh-cli/             # 管理 CLI（P1 可最小化）

internal/                  # 核心业务实现（不可被外部项目导入）
  config/                  # 配置加载与校验
  net/                     # 网络监听与 TLS/mTLS 设置
  security/                # PKI 与证书管理（基于 Go 标准库，无 OpenSSL 依赖）
  session/
    model.go               # Session/Token 数据结构
    index.go               # 多维索引实现
    store.go               # 内存存储 + 持久化适配
    lifecycle.go           # 生命周期操作核心逻辑
  persistence/
    wal.go                 # WAL 写入与回放
    snapshot.go            # 快照读写
  resources/
    memlimit.go            # 内存阈值检测与策略
  api/
    httpapi/               # HTTP API handlers
  admin/                   # 管理 API（独立端口）

docs/                      # 设计文档（SDD）与规范
  sdd/                     # 项目设计文档
  standards/sdd/           # SDD 标准与 AI 协作规范
```

### 关键架构模式

**双端口架构**（R19 安全域隔离）：
- **业务数据平面**（默认 :8080）：
  - 服务 SSO/IAM 和 App 客户端
  - 会话生命周期 API（创建/校验/续期/撤销）
  - 支持 TLS/mTLS（可选）
- **管理控制平面**（默认 :8081）：
  - 仅供 tokmesh-cli / IAM 运维使用
  - 证书管理、统计诊断、手工踢人
  - **生产环境强制 mTLS**

**Session/Token 关系模型**：
- Session（会话）：主对象，包含用户标识、设备信息、登录 IP、租户、时间戳、状态等
- Token（令牌）：附属凭证，一个会话下可有多个不同类型令牌（access/refresh/admin）
- 多维索引作用于 Session 层，支持按 User/Device/Tenant 快速查询与踢人

**阶段性演进路线**：
- **P1（当前）**：MVP + mTLS（单节点、基础持久化、核心安全）
- **P2**：安全增强（加密、主动防御、App 鉴权）
- **P3**：集群化（分布式哈希环、副本、Any-Node Gateway）
- **P4**：SDK 生态（多语言客户端库）

## 文档驱动开发（Doc-First）—— 最重要约束

### 铁律：先写文档再写代码

**在 TokMesh-Go 项目中，文档优先是不可突破的铁律**：

1. **没有对应的 SDD 记录，就不应该有代码变更**
2. **任何新特性、重大改动、关键 bug 修复，在编码前必须至少完成 SDD-1（规格/冻结）阶段**
3. **代码合并前，必须可以从 SDD-2 追溯到 SDD-1 中已冻结的 RQ-ID**

### SDD 三阶段晋级制度

**SDD-0：探索（Exploration）**
- 目标：明确"我们在做什么 / 为什么要做"
- 内容：问题背景、使用场景、候选方案对比、TokMesh 技术约束识别
- 输出：为 SDD-1 提供足够信息，但不进入实现细节

**SDD-1：规格/冻结（Specification / Freeze）**
- 目标：明确"系统应该表现成什么样子"，是需求的**冻结点**
- 内容：
  - RQ-ID 列表（如 R1, R2, ...），每个 RQ 包含描述、触发条件、期望行为、约束条件
  - 绑定 TokMesh 技术约束
  - 非功能需求（性能指标、可靠性目标、可观察性）
  - 验收标准（如何在测试中验证）
- 原则：**代码不得超出 SDD-1 中已冻结的 RQ-ID 范围**

**SDD-2：实现/交付（Implementation / Delivery）**
- 目标：明确"系统将如何被实现"，设计与实现之间的桥梁
- 内容：
  - 模块/包结构设计
  - 接口与数据模型
  - RQ-ID → 实现映射（哪些模块负责哪些需求）
  - 测试计划（覆盖率目标 > 80%）
  - Benchmark 计划（关键路径性能指标）
  - 部署与运维要点
- 原则：**所有代码变更必须可追溯到 SDD-2 中的设计决策**

## AI 协作规则（基于 00-constitution.md）

### 双重角色

**虚拟架构师（Virtual Architect）**：
- 协助需求澄清、场景建模、架构选型、模块拆分、接口设计
- 确保所有架构决策有明确动机和可验证结果
- **在 SDD 文档完成或更新之前，不得直接给出代码实现方案**

**合规审查员（Compliance Reviewer）**：
- 评估需求、设计、代码是否符合文档治国原则、SDD 晋级制度、TokMesh 技术约束、质量红线
- 给出"合规 / 不合规 / 部分合规"结论，并指出违反条款、具体问题、修正建议
- **敢于说"不"**：在不满足宪法要求时，不得默认认可，而应明确标记并要求整改

### 对"绕过文档阶段"的处理

当用户试图绕过文档阶段直接请求写代码时，AI 必须：
1. 明确声明当前请求**不符合文档治国原则**
2. 概述缺失哪些 SDD 阶段（SDD-0 / SDD-1 / SDD-2）
3. 提供最小成本的补文档路径
4. 如用户坚持绕过，可在用户明确知情前提下提供建议，但必须标记为**"非合规建议（仅供草拟/探索）"**

## 更新想法与需求的标准流程

### 零散想法 → 正式需求的路径

当用户提出新想法或功能请求时，AI 应按以下流程处理：

**第一步：收入 Backlog Inbox**
- 将零散想法先记入 `docs/sdd/TokMesh-v1-04-backlog-inbox.md`
- 使用格式：`INBOX-YYYYMMDD-XX：简短标题`
- 标记状态为 `pending`
- 初步判断可能关联的现有 R 编号或需要新增 R

**第二步：虚拟架构师整理与评估**
- 分析想法是否属于现有 R（R1-R29）的补充
- 如是补充：直接更新 `docs/sdd/TokMesh-v1-02-requirements.md` 中对应的 R 条目
- 如是新需求：分配新的 R 编号（如 R30），添加到需求清单
- 将 Backlog Inbox 中对应条目标记为 `merged`

**第三步：SDD-0 探索（如需要）**
- 对于复杂或有多种实现方案的需求，需要先完成 SDD-0 探索
- 输出内容：问题背景、使用场景、候选方案、TokMesh 约束识别、开放问题
- 可记录在主题设计文档中（如 `TokMesh-v1-XX-design-*.md`）

**第四步：SDD-1 规格冻结**
- 明确需求的触发条件、期望行为、约束条件、验收标准
- 在 `docs/sdd/TokMesh-v1-02-requirements.md` 中完善 R 条目细节
- 或在主题设计文档中详细展开（引用 R 编号）

**第五步：SDD-2 实现设计（编码前）**
- 模块/包结构设计、接口签名、数据模型
- R 编号 → 实现映射
- 测试计划（覆盖率目标）、Benchmark 计划
- 当前主要承载在 `docs/sdd/TokMesh-v1-30-impl-p1-blueprint.md`

## 质量红线

### 性能优先（必须有 Benchmark）

- 对所有**关键路径**和**核心算法**，必须在 SDD-1 中给出明确性能指标（P95 延迟、最小吞吐、资源占用上限）
- 在 SDD-2 中，必须为关键路径规划对应的 benchmark（指明要在哪个包中编写 `BenchmarkXxx`，观测哪些指标）
- 编码时，AI 必须：
  - 当引入新核心逻辑时，主动建议相应的 benchmark 函数设计
  - 若用户未提供 benchmark，明确提示"性能红线未满足"，并引导补全
- **若无合理理由缺失 benchmark，合规审查员必须标记为不合规**

### 测试覆盖率 > 80%

- SDD-2 中必须声明测试覆盖目标（模块级或整体级）
- 测试设计应以 RQ-ID 为主线，构建测试用例矩阵：
  - 每个 RQ-ID 至少对应一组正向与一组异常场景测试
  - 对关键错误路径、边界条件、并发场景进行重点测试
- AI 编写代码时，应尽量同步提供对应的测试建议或测试样例
- 使用 `go test ./... -cover` 评估覆盖率
- **若覆盖率无法合理接近或达到 80%，合规审查员必须标记为不合规或部分合规**

### 开发与测试节奏（小步前进）

- 在 TokMesh 项目中，Claude 参与编码/重构时，应遵循“小步实现 + 立即测试”的基本节奏，而不是“先写完一大块再集中补测”：  
  - 每完成一个小能力块（例如一个内部包的一组核心函数、一个 HTTP handler、一个 gRPC service 的基础骨架），应尽快补充并运行相应的单元测试或最小集成测试；  
  - 对关键路径与高风险改动，应优先增加针对性的测试用例，而不是依赖后续大而全的测试补齐；  
  - 在对已有模块进行修改时，应优先理解现有测试（如有），并在此基础上增加回归测试，而不是直接替换行为。  
- 开发阶段不要求每次微小改动都运行全量测试，但在以下场景中，Claude 应主动建议或执行 `go test ./...`：  
  - 合并前的最终检查；  
  - 对核心模块（session、persistence、cluster、安全基线等）进行非小修的改动；  
  - 在 SDD-2 或实现蓝图中标记为“质量红线”的路径（如 ValidateToken）被改动时。  

## TokMesh 特有技术约束

### 语言与运行时约束
- Go 版本：`go 1.21+`（见 `go.mod`）
- 必须使用 Go modules 进行依赖管理
- PKI 实现必须基于 Go 标准库，**不依赖 OpenSSL**

### 协议与通信约束
- 双端口架构强制要求（业务端口 + 管理端口分离）
- 管理端口生产环境强制 mTLS
- Session/Token 是核心领域对象，存储与协议设计应以"会话为核心，令牌为附属凭证"思路展开

### 性能与扩展性约束
- 针对高频操作（ValidateToken、CreateSession、ExtendSession）必须有性能基线
- 内存阈值管理（见 R12）：达到阈值时新建写入返回错误，保护稳定性

### 安全与隔离约束
- 零信任网络假设（不假设内网安全）
- 业务/管理 API 必须严格端口隔离
- 敏感数据处理需遵循 P2 加密设计草案（当前 P1 阶段预留钩子）

**AI 行为要求**：
- 在提出任何方案时，必须检查是否违反已声明的 TokMesh 约束
- 如存在潜在冲突，应明确指出冲突点、风险、折中方案
- 在不确定约束细节时，**不得假定"没有约束"**，应向用户确认或建议在 SDD 中补充

## 合规审查输出格式建议

```
- ...
```

## 快速参考：AI 在不同场景下的操作流程

### 场景 1：用户提出新想法或功能请求

```
1. 检查是否已有相关 R 编号
   → 如有：更新对应 R 的描述
   → 如无：添加到 Backlog Inbox

2. 作为虚拟架构师评估复杂度
   → 简单补充：直接更新需求清单
   → 复杂需求：引导完成 SDD-0 探索

3. 绝不直接写代码
```

### 场景 2：用户要求"实现某功能"或"写代码"

```
1. 执行检查清单（见上文"AI 必须遵循的检查清单"）
   ☐ 是否有 R 编号？
   ☐ 是否有 SDD-1 规格？
   ☐ 是否有 SDD-2 设计？

2. 如任一缺失
   → 明确声明不符合文档治国原则
   → 指导用户补齐缺失阶段

3. 仅在全部满足时
   → 编写代码
   → 同步提供测试
   → 实现 benchmark（如需要）
```

### 场景 3：用户提出重构或优化

```
1. 作为虚拟架构师评估
   → 是否违反 SDD-2 设计边界？
   → 是否影响已有 R 的语义？

2. 如涉及架构变更
   → 先更新 SDD-2
   → 再实施重构

3. 如仅优化实现
   → 确保测试覆盖不降低
   → 更新相关 benchmark
```

### 场景 4：用户要求审查代码或设计

```
1. 作为合规审查员，按三维度评估
   ☐ SDD 流程合规性
   ☐ TokMesh 技术约束
   ☐ 质量红线（性能、测试覆盖率）

2. 输出结构化结论
   → 合规 / 部分合规 / 不合规
   → 问题列表
   → 建议的下一步
```

### 场景 5：遇到 Backlog Inbox 中的议题

```
1. 先查阅 TokMesh-v1-04-backlog-inbox.md
   → 了解已有讨论与开放问题

2. 避免重复讨论
   → 直接引用 INBOX-XXX 编号
   → 补充新见解或推进决策

3. 如议题已解决
   → 更新对应 R 编号
   → 标记 Backlog 条目为 merged
```

### 关键禁止项

❌ **绝不**在没有 SDD-1 的情况下编写代码
❌ **绝不**假设"没有文档就是没有约束"
❌ **绝不**在质量红线（测试覆盖率 < 80%、缺失 benchmark）不满足时标记为"合规"
❌ **绝不**绕过 Backlog Inbox 直接将未经评估的想法添加到需求清单
❌ **绝不**在 P1 阶段实现 P2/P3/P4 功能的完整代码（可预留钩子）
