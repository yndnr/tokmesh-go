# 想法: 全链路安全防护

状态: 草稿
创建日期: 2025-12-05
编号: CP-254903
关联文档: CP-254901

## 原始想法

TokMesh 采用**安全优先 + 纵深防御**策略，提供全链路安全防护（内存 + 落盘 + 传输 + Web 安全 + 攻击防护），在保证安全的前提下追求极致性能。

## 内存与落盘加密

### 内存加密与脱敏

#### 敏感字段加密
- **对敏感数据进行内存级加密存储**
- 即使内存被dump，敏感数据也无法直接读取
- 使用高性能加密算法（如 AES-GCM），最小化性能影响

#### 脱敏存储策略

1. **令牌存储**：
   - 仅保存令牌摘要（Hash，如 SHA-256）
   - 不存储完整明文令牌
   - 校验时对比摘要

2. **用户敏感信息**：
   - 手机号部分隐藏（如 138****1234）
   - 邮箱部分隐藏（如 u***r@example.com）
   - IP 地址可选脱敏

3. **性能与安全平衡**：
   - 在保证 P99 < 1ms 的前提下实现强制加密
   - 使用硬件加速（AES-NI）降低加密开销

### 落盘数据加密

#### 快照加密
- **定期快照文件全盘加密**
- 支持多种加密算法（AES-256、国密 SM4）
- 快照文件使用独立密钥

#### WAL 日志加密
- **追加写日志加密存储**
- 实时加密，避免明文落盘
- 支持日志文件轮转时密钥更新

#### 密钥管理

1. **密钥分级**：
   - 主密钥（Master Key）：加密数据密钥
   - 数据密钥（Data Encryption Key, DEK）：加密实际数据
   - 会话密钥：加密传输层数据

2. **密钥轮换**：
   - 支持定期自动轮换密钥
   - 轮换时不影响在线服务
   - 旧数据逐步重新加密

3. **密钥存储**：
   - 支持外部密钥管理系统（KMS）集成
   - 支持硬件安全模块（HSM）
   - 支持本地密钥文件（开发环境）

## Web 安全与攻击防护

### 设计原则

TokMesh 本身不直接负责前端渲染，但在协议和数据模型层面为 Web 安全提供支持，**避免成为攻击放大器**。

### CSRF（跨站请求伪造）防护支撑

#### 上层系统建议策略
- 使用 `HttpOnly + Secure + SameSite` Cookie 存放敏感会话标识
- 使用独立的 CSRF Token（与 TokMesh 中的会话/令牌 ID 绑定）

#### TokMesh 数据模型支持

1. **预留字段**：
   - 记录 "CSRF Token ID / 版本号" 元数据
   - SSO/网关校验时可携带并比对 CSRF Token

2. **高风险 Token 的 TTL 缩短**：
   - 对跨站环境下使用的短 Session 使用更短的 TTL
   - 降低被劫持后的风险窗口

#### API 访问策略

- **管理 API 与业务 API 默认不支持浏览器跨站调用**
- 默认不通过 CORS 暴露给任意前端
- 建议仅由后端服务（SSO/网关/微服务）通过服务器间调用访问

### XSS（跨站脚本攻击）防护

#### 响应格式安全
- **仅返回结构化数据**（JSON / Protobuf），不生成 HTML
- 不拼接前端可执行代码，从根源上降低 XSS 风险

#### 存储数据安全

1. **输出编码要求**：
   - 对可显示字段（用户代理、终端描述、备注等）要求：
   - 统一进行输出编码（HTML 转义 / 前端模板安全输出）
   - 规范在 governance 或 UI 侧文档中明确

2. **避免富文本**：
   - 在接口设计上避免将任意未校验的富文本直接返回给管理前端

### 重放攻击（Replay Attack）防护

#### 管理 API 防护（多层防御）

**第 1 层：时间戳 + 随机数（Nonce）**
- 所有敏感写操作（踢人、吊销、策略变更、证书分发）要求：
  - 携带时间戳（timestamp）
  - 携带随机数（nonce）
- 服务端维护短期 nonce 使用记录（按调用方身份 + nonce）
- 在时间窗口内拒绝重复使用的 nonce 请求

**第 2 层：签名验证**
- 携带基于密钥的签名（HMAC / 国密签名）
- 或依赖 mTLS 并结合服务端侧的 nonce 校验
- 签名包含请求体、时间戳、nonce

**第 3 层：幂等键（Idempotency Key）**
- 支持 `idempotency_key`
- 防止网络重试导致的操作重复执行
- 服务端记录已处理的幂等键

**第 4 层：mTLS 增强（客户端身份绑定）**
- 对管理调用方规定**固定的客户端证书身份**
- 与 API Key / 签名机制**叠加使用**（多因子认证）
- 证书 Subject DN / CN 与调用方身份强绑定
- **进一步降低重放风险**：即使签名被截获，没有对应的客户端证书也无法重放

**mTLS 防护优势**：
- 证书难以伪造和窃取
- 证书可设置短期有效期并支持吊销
- 与 nonce + 签名机制形成纵深防御

#### 业务 API 防护

1. **令牌设计要求**：
   - 令牌包含唯一 ID（jti）
   - 签发时间（iat）
   - 过期时间（exp）

2. **使用模式检测**：
   - 可选支持"令牌使用计数 / 最近一次使用时间"记录
   - 识别异常的重复使用模式

3. **异常检测与告警**：
   - 对同一令牌在极短时间内来自异常 IP 的重复校验请求：
     - 结合速率限制与安全日志进行告警
     - 可触发自动封禁或人工审核

## 速率限制与滥用控制

### 可配置的速率限制

#### 按调用方标识限流
- **客户端证书**（mTLS 场景）
- **API Key**
- **IP 地址**
- **租户 ID**（多租户场景）

#### 管理 API 严格限流

对敏感操作配置更严格的限流：
- **踢人操作**：如 10 次/分钟
- **吊销操作**：如 5 次/分钟
- **策略变更**：如 2 次/分钟
- **批量操作**：如 1 次/10分钟

### 职责划分

#### 本服务内部：保护型限流
- 简单的"保护型限流"（防止被暴力刷爆）
- 基于令牌桶或漏桶算法
- 快速失败，不影响正常流量

#### 外部网关协作：策略性限流
- 大规模策略性限流可交由 API 网关 / WAF / 安全网关处理
- 支持更复杂的限流策略（如地理位置、用户行为分析）
- 本服务提供限流状态查询接口

## 日志与隐私保护

### 敏感字段保护

#### 不记录高敏感字段
- 日志中**不记录完整令牌/密钥**等高敏感字段
- 使用**部分截断或哈希**代替明文
  - 令牌：仅记录前 8 字符 + 哈希
  - 密钥：完全不记录
- 避免记录用户密码、完整 Cookie 等

#### 脱敏规则示例

```
# 原始数据
token: "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIiwiaWF0IjoxNTE2MjM5MDIyfQ.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c"

# 日志记录
token: "eyJhbGci... (sha256: a3f2...)"
```

### 最小必要原则

#### 安全分析用途标记
- IP、User-Agent、设备 ID 等标记为"安全分析用途"
- 在日志与导出中遵循最小必要原则
- 仅在安全事件调查时完整记录

#### 管理日志参数记录
- 管理日志参数记录时避免包含敏感信息
- 示例：记录"用户 ID: 12345"而非"密码: xxx"

### 审计日志完整性

#### 审计日志要求
- **不可篡改**：使用只写存储或区块链技术
- **完整性校验**：日志文件签名或哈希链
- **长期保存**：满足合规要求（如 1-7 年）

#### 审计内容

记录以下操作：
- 会话创建 / 销毁
- 令牌签发 / 吊销
- 管理操作（踢人、策略变更）
- 认证失败 / 授权拒绝
- 配置变更
- 密钥轮换

## 补充说明

### 安全策略总结

**安全优先 + 纵深防御**：

1. **数据保护**：
   - 内存加密 + 落盘加密 + 传输加密
   - 敏感字段脱敏
   - 密钥分级管理

2. **Web 安全**：
   - CSRF 防护支撑
   - XSS 防护（仅返回结构化数据）
   - API 访问策略（不暴露给浏览器）

3. **攻击防护**：
   - 重放攻击防护（4 层防御）
   - 速率限制与滥用控制
   - 异常检测与告警

4. **隐私保护**：
   - 日志脱敏
   - 最小必要原则
   - 审计日志完整性

### 合规支持

支持以下合规标准：
- **GDPR**（欧盟通用数据保护条例）：数据脱敏、最小必要原则
- **等保 2.0**（中国信息安全等级保护）：访问控制、审计、加密
- **SOC 2**：安全、可用性、处理完整性
- **ISO 27001**：信息安全管理体系
