# 想法: 运维能力

状态: 草稿
创建日期: 2025-12-05
编号: CP-254905
关联文档: CP-254901

## 原始想法

TokMesh 提供**完整的运维能力**：Prometheus 原生可观测性 + 速率限制与滥用控制 + 日志与隐私保护 + CLI 管理工具，让系统易于监控、管理和故障排查。

## 监控与可观测性（Prometheus 集成）

### Prometheus Metrics 端点

#### 标准端点支持

**HTTP 端点**：
- 提供 `/metrics` 端点（路径和端口可配置）
- 供 Prometheus 拉取（Pull Model）
- 输出格式：Prometheus 文本格式或 OpenMetrics 格式

**配置示例**：
```yaml
monitoring:
  prometheus:
    enabled: true
    path: /metrics        # 可配置路径
    port: 9100            # 独立端口（可选）
    address: 127.0.0.1    # 监听地址
```

#### 访问控制

**内网访问限制**：
- 绑定内网 IP（如 `127.0.0.1`, `10.0.0.0/8`）
- 避免暴露到公网

**白名单支持**：
- IP 白名单（如 `192.168.1.0/24`）
- CIDR 列表

**mTLS 认证**：
- 客户端证书验证
- 仅允许持有有效证书的 Prometheus 实例访问
- 配置示例：
  ```yaml
  monitoring:
    prometheus:
      tls:
        enabled: true
        cert_file: /etc/tls/server.crt
        key_file: /etc/tls/server.key
        ca_file: /etc/tls/ca.crt
        client_auth: require
  ```

**避免暴露敏感监控数据**：
- 监控端点不暴露令牌、密钥等敏感信息
- 仅暴露聚合指标和统计数据

### 指标覆盖维度

#### 1. 性能指标

**QPS（每秒请求数）**：
- `tokmesh_api_requests_total`（Counter）
- 按 API 类型分类：`{api="business"}`, `{api="admin"}`

**延迟分布**：
- `tokmesh_api_request_duration_seconds`（Histogram）
- 分位数：P50, P90, P95, P99, P999

**缓存命中率**：
- `tokmesh_cache_hits_total`（Counter）
- `tokmesh_cache_misses_total`（Counter）

**吞吐量**：
- `tokmesh_bytes_in_total`（Counter）
- `tokmesh_bytes_out_total`（Counter）

#### 2. 容量指标

**在线会话数**：
- `tokmesh_sessions_active`（Gauge）
- 按会话类型分类：`{type="web"}`, `{type="api"}`

**令牌总数**：
- `tokmesh_tokens_total`（Gauge）
- 按令牌类型分类：`{type="access"}`, `{type="refresh"}`

**内存使用率**：
- `tokmesh_memory_usage_bytes`（Gauge）
- `tokmesh_memory_usage_percent`（Gauge）

**存储使用率**：
- `tokmesh_storage_usage_bytes`（Gauge）
- `tokmesh_storage_capacity_bytes`（Gauge）

**连接数**：
- `tokmesh_connections_active`（Gauge）
- `tokmesh_connections_idle`（Gauge）

#### 3. 集群状态

**节点健康状态**：
- `tokmesh_node_health`（Gauge）：1 = 健康, 0 = 不健康

**主从角色**：
- `tokmesh_node_role`（Gauge）：1 = Leader, 0 = Follower

**Raft 集群状态**：
- `tokmesh_raft_term`（Gauge）：当前任期
- `tokmesh_raft_commit_index`（Gauge）：已提交索引
- `tokmesh_raft_applied_index`（Gauge）：已应用索引

**数据同步延迟**：
- `tokmesh_replication_lag_seconds`（Gauge）

#### 4. 安全与错误

**认证失败次数**：
- `tokmesh_auth_failures_total`（Counter）
- 按原因分类：`{reason="invalid_token"}`, `{reason="expired"}`

**授权拒绝次数**：
- `tokmesh_authz_denials_total`（Counter）

**速率限制触发次数**：
- `tokmesh_ratelimit_hits_total`（Counter）
- 按调用方分类：`{caller="client_1"}`

**重放攻击检测次数**：
- `tokmesh_replay_attacks_detected_total`（Counter）

**错误率**：
- `tokmesh_errors_total`（Counter）
- 按错误类型分类：`{type="timeout"}`, `{type="internal_error"}`

### Prometheus 依赖策略

#### 优先使用成熟 client 库
- 优先采用官方 Prometheus Go client（`github.com/prometheus/client_golang`）

#### 轻量化实现考虑

若依赖体积过大且只用少量功能，考虑**自研轻量实现**：
- 自己维护 `Counter` / `Gauge` / `Histogram` / `Summary` 等基础类型
- 输出 Prometheus 文本格式或 OpenMetrics 格式
- **保持可观测性与产物体积之间的平衡**

**文本格式示例**：
```
# HELP tokmesh_api_requests_total Total API requests
# TYPE tokmesh_api_requests_total counter
tokmesh_api_requests_total{api="business"} 12345
tokmesh_api_requests_total{api="admin"} 678
```

### CLI 基础指标查看

**CLI 命令**：
- `tokmesh-cli metrics show`：查看实时指标
- `tokmesh-cli metrics export`：导出指标到文件

**示例输出**：
```
TokMesh Metrics Summary
=======================
QPS:               12,345 req/s
P99 Latency:       0.8 ms
Active Sessions:   54,321
Cache Hit Rate:    98.5%
Memory Usage:      2.3 GB / 8 GB (28.75%)
```

**职责划分**：
- CLI 提供基础指标查看（只读模式）
- **复杂告警与展示交由 Prometheus 生态完成**（Grafana / Alertmanager）

## 速率限制与滥用控制（补充）

### 可配置的速率限制

#### 按调用方标识限流
- **客户端证书**（mTLS 场景）
- **API Key**
- **IP 地址**
- **租户 ID**（多租户场景）

#### 管理 API 严格限流

对敏感操作配置更严格的限流：
```yaml
ratelimit:
  admin_api:
    kick_user:      # 踢人操作
      limit: 10
      window: 1m
    revoke_token:   # 吊销操作
      limit: 5
      window: 1m
    update_policy:  # 策略变更
      limit: 2
      window: 1m
    batch_operation: # 批量操作
      limit: 1
      window: 10m
```

### 职责划分

**本服务内部**：
- 简单的"保护型限流"（防止被暴力刷爆）
- 基于令牌桶或漏桶算法
- 快速失败，不影响正常流量

**外部网关协作**：
- 大规模策略性限流可交由 **API 网关 / WAF / 安全网关**处理
- 支持更复杂的限流策略（如地理位置、用户行为分析）
- TokMesh 提供限流状态查询接口

## 日志管理

### 日志级别

支持标准日志级别：
- **DEBUG**：调试信息
- **INFO**：正常操作信息
- **WARN**：警告信息
- **ERROR**：错误信息
- **FATAL**：致命错误，导致服务退出

### 结构化日志

**JSON 格式**：
```json
{
  "timestamp": "2025-12-05T10:30:45Z",
  "level": "INFO",
  "module": "auth",
  "message": "Token validated successfully",
  "user_id": "12345",
  "session_id": "abc123",
  "ip": "192.168.1.100",
  "latency_ms": 0.5
}
```

**好处**：
- 易于解析和查询
- 支持日志聚合工具（ELK、Loki、Splunk）
- 支持字段过滤和搜索

### 日志脱敏（重申）

#### 敏感字段保护
- 日志中**不记录完整令牌/密钥**
- 使用**部分截断或哈希**代替明文
- 避免记录用户密码、完整 Cookie

#### 脱敏规则
```
# 原始数据
token: "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."

# 日志记录
token: "eyJhbGci... (sha256: a3f2...)"
```

### 日志轮转

**按大小轮转**：
- 日志文件达到指定大小（如 100 MB）时轮转
- 保留最近 N 个文件

**按时间轮转**：
- 每天或每小时创建新日志文件
- 示例：`tokmesh-2025-12-05.log`

**压缩旧日志**：
- 自动压缩旧日志文件（gzip）
- 节省存储空间

**配置示例**：
```yaml
logging:
  level: INFO
  format: json
  output: /var/log/tokmesh/tokmesh.log
  rotation:
    max_size: 100MB
    max_age: 30d
    max_backups: 10
    compress: true
```

## 配置管理

### 配置来源

**多种配置来源**：
1. **配置文件**：YAML / JSON / TOML
2. **环境变量**：覆盖配置文件
3. **命令行参数**：覆盖环境变量
4. **远程配置中心**（可选）：Consul / etcd / Nacos

**优先级**：
```
命令行参数 > 环境变量 > 配置文件 > 默认值
```

### 配置热重载

**支持配置热重载**：
- 监听配置文件变化
- 自动重新加载配置
- 无需重启服务

**不支持热重载的配置**：
- 监听地址和端口（需要重启）
- TLS 证书路径（需要重启）

**配置示例**：
```yaml
config:
  reload:
    enabled: true
    watch_interval: 10s
```

### 配置验证

**启动时验证**：
- 启动前验证配置文件格式和内容
- 提供清晰的错误提示
- 验证失败时拒绝启动

**示例**：
```bash
$ tokmesh --config /etc/tokmesh/config.yaml --validate
[OK] Configuration is valid
```

## 审计日志（重申）

### 审计内容

记录以下操作：
- **会话创建 / 销毁**
- **令牌签发 / 吊销**
- **管理操作**（踢人、策略变更）
- **认证失败 / 授权拒绝**
- **配置变更**
- **密钥轮换**

### 审计日志要求

**不可篡改**：
- 使用只写存储或区块链技术
- 审计日志单独存储，不与业务日志混合

**完整性校验**：
- 日志文件签名或哈希链
- 定期验证日志完整性

**长期保存**：
- 满足合规要求（如 1-7 年）
- 支持归档到对象存储（S3、OSS）

### 审计日志格式

```json
{
  "timestamp": "2025-12-05T10:30:45Z",
  "event_type": "session_created",
  "user_id": "12345",
  "session_id": "abc123",
  "ip": "192.168.1.100",
  "user_agent": "Mozilla/5.0...",
  "result": "success",
  "operator": "system"
}
```

## 健康检查

### Health Check 端点

**多种探针**：
- **Liveness Probe**：`GET /health/live`
  - 检测服务是否存活
  - 失败时 Kubernetes 重启 Pod

- **Readiness Probe**：`GET /health/ready`
  - 检测服务是否准备好接收流量
  - 失败时从 Service 中移除 Pod

- **Startup Probe**：`GET /health/startup`
  - 检测服务是否启动完成
  - 适用于启动较慢的服务

**检查内容**：
- 数据库连接（如果有）
- 集群节点通信
- 内存使用率
- 磁盘空间

**响应示例**：
```json
{
  "status": "healthy",
  "checks": {
    "cluster": "ok",
    "memory": "ok",
    "disk": "ok"
  },
  "timestamp": "2025-12-05T10:30:45Z"
}
```

## CLI 管理工具

### 常用命令

**会话管理**：
```bash
# 查看会话列表
tokmesh-cli session list --user-id 12345

# 查看会话详情
tokmesh-cli session get abc123

# 踢出会话
tokmesh-cli session kick abc123
```

**令牌管理**：
```bash
# 查看令牌列表
tokmesh-cli token list --user-id 12345

# 吊销令牌
tokmesh-cli token revoke xyz789
```

**统计信息**：
```bash
# 查看在线用户统计
tokmesh-cli stats sessions --group-by device_type

# 查看令牌统计
tokmesh-cli stats tokens --group-by token_type
```

**配置管理**：
```bash
# 查看当前配置
tokmesh-cli config get

# 更新配置
tokmesh-cli config set ratelimit.admin_api.kick_user.limit 20

# 重新加载配置
tokmesh-cli config reload
```

- 结构化日志（JSON）

- 多种配置来源

- 完整的审计日志
- 不可篡改性
- 长期保存

**管理**：
- CLI 管理工具
- 健康检查端点

   - OpenTelemetry 集成
   - 分布式追踪
