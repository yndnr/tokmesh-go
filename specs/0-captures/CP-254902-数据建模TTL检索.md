# 想法: 数据建模、TTL 策略与检索能力

状态: 草稿
创建日期: 2025-12-05
编号: CP-254902
关联文档: CP-254901

## 原始想法

TokMesh 相比 Redis 的核心差异化能力：**多维度数据建模 + 策略化 TTL 管理 + 受控复杂度的检索能力**。不是简单的 KV 存储，而是针对会话/令牌场景的专用数据模型。

## 数据维度建模

### 支持的六大维度

#### 1. 用户维度
- **用户 ID**（必需）：唯一标识符
- **账号**：用户名 / 邮箱 / 手机号
- **用户类型**：普通用户 / VIP 用户 / 管理员等

**典型查询**：
- 查询某用户的所有会话
- 按用户类型统计在线数

#### 2. IP 维度
- **IPv4 / IPv6**：完整支持
- **单 IP 查询**：精确匹配
- **网段查询**：支持 CIDR（如 192.168.1.0/24）
- **前缀查询**：支持前缀匹配

**典型查询**：
- 查询来自某 IP 的所有会话
- 查询来自某网段的活跃令牌数
- 检测异常 IP 的访问模式

#### 3. 会话维度
- **会话 ID**（必需）：唯一标识符
- **状态**：活跃 / 过期 / 已吊销
- **创建时间**：会话建立时间
- **最后活跃时间**：最近一次访问时间
- **会话类型**：Web 会话 / API 会话 / 管理会话

**典型查询**：
- 查询某会话的详细信息
- 查询超过 N 小时未活跃的会话
- 按会话类型统计分布

#### 4. 令牌维度
- **令牌 ID**（jti，必需）：唯一标识符
- **令牌类型**：访问令牌 / 刷新令牌 / 管理令牌
- **作用域（Scope）**：权限范围
- **签发方（Issuer）**：令牌签发者
- **签发时间（iat）**：令牌生成时间
- **过期时间（exp）**：令牌失效时间

**典型查询**：
- 查询某令牌的状态
- 查询某作用域下的所有令牌
- 按令牌类型统计分布

#### 5. 终端维度
- **终端类型**：Web / Mobile / Desktop / API Client / IoT
- **设备标识**：设备唯一 ID（可选）
- **操作系统**：Windows / Linux / macOS / iOS / Android
- **浏览器信息**（可选）：User-Agent 解析后的浏览器类型和版本

**典型查询**：
- 查询某用户在所有设备上的会话
- 查询某终端类型（如 Mobile）的在线用户数
- 检测异常的设备登录

#### 6. 类型维度
- **会话类型**：用于差异化 TTL 策略
- **令牌类型**：用于差异化安全策略
- **终端类型**：用于差异化访问控制

**典型查询**：
- 按会话类型、令牌类型、终端类型进行统计
- 为不同类型配置不同的 TTL 和限流策略

### 设计约束

#### 受控的查询复杂度
- ✅ 支持：单维度查询、多维度组合查询（最多 3 个维度）
- ❌ 拒绝：任意字段的全文检索、正则表达式查询、复杂的 JOIN 操作
- ✅ 保证：所有查询的复杂度可预测，性能不受数据量线性影响

#### 索引策略
- 主键索引：会话 ID、令牌 ID（哈希索引）
- 二级索引：用户 ID、IP、终端类型等（B-Tree 或倒排索引）
- 复合索引：用户 ID + 终端类型、IP + 会话类型等
- 索引更新：实时更新，保证查询一致性

## TTL 与过期策略

### 多种 TTL 模型

#### 1. 固定 TTL（Fixed TTL）
- **行为**：创建时设定，到期自动失效
- **适用场景**：
  - 访问令牌（短期有效，如 5 分钟）
  - 一次性会话（临时访问）
- **配置示例**：
  ```yaml
  token:
    access_token:
      ttl: 300s  # 5 分钟
  ```

#### 2. 滑动 TTL（Sliding / Rolling TTL）
- **行为**：每次访问自动延长过期时间
- **适用场景**：
  - Web 会话（用户活跃时保持登录）
  - 移动端长会话
- **配置示例**：
  ```yaml
  session:
    web_session:
      ttl: 3600s      # 基础 TTL 1 小时
      sliding: true   # 启用滑动
      max_ttl: 86400s # 最大 TTL 24 小时
  ```

#### 3. 分级 TTL（Tiered TTL）
- **行为**：不同会话/令牌类型使用不同 TTL
- **适用场景**：
  - VIP 用户会话更长
  - 管理令牌更短（安全考虑）
  - 刷新令牌更长（如 7 天）
- **配置示例**：
  ```yaml
  session:
    normal_user: 3600s   # 普通用户 1 小时
    vip_user: 7200s      # VIP 用户 2 小时
    admin_user: 1800s    # 管理员 30 分钟（安全优先）
  token:
    access_token: 300s   # 访问令牌 5 分钟
    refresh_token: 604800s # 刷新令牌 7 天
  ```

### 策略化管理

#### 差异化 TTL 配置

支持按以下维度配置不同的 TTL：

1. **用户类型**：
   - 普通用户：标准 TTL
   - VIP 用户：延长 TTL
   - 管理员：缩短 TTL（安全优先）

2. **终端类型**：
   - Web：固定 TTL 或 Sliding TTL
   - Mobile：Sliding TTL（用户体验优先）
   - API Client：固定 TTL（安全优先）
   - Desktop：Sliding TTL

3. **令牌类型**：
   - 访问令牌：短 TTL（5-15 分钟）
   - 刷新令牌：长 TTL（7-30 天）
   - 管理令牌：极短 TTL（1-5 分钟）

4. **租户 / 组织**（多租户场景）：
   - 不同租户可配置不同的 TTL 策略
   - 支持租户级别的策略继承和覆盖

### 渐进式过期清理

#### 问题：集中过期导致的性能抖动

大规模会话同时过期会导致：
- 突发的删除操作，CPU 峰值
- 大量用户同时重新登录，认证系统压力激增
- 缓存雪崩效应

#### 解决方案：渐进式淘汰

1. **周期性清理**：
   - 后台线程定期扫描过期数据
   - 每次清理一小批（如 1000 条）
   - 清理间隔可配置（如 100ms）

2. **懒惰删除**：
   - 访问时检查是否过期
   - 过期则立即删除并返回"未找到"
   - 减少主动扫描的压力

3. **随机 TTL 抖动**：
   - 在设定的 TTL 基础上增加随机偏移（如 ±5%）
   - 避免大量会话同时过期
   - 示例：3600s TTL → 3420s ~ 3780s

4. **优雅降级**：
   - 过期清理速率受控，不影响在线业务
   - 高负载时降低清理频率
   - 保证业务 API 优先

## 检索能力

### 支持的查询类型

#### 1. 单维度查询
```
# 按用户 ID 查询
GET /sessions?user_id=12345

# 按 IP 查询
GET /sessions?ip=192.168.1.100

# 按终端类型查询
GET /sessions?device_type=mobile
```

#### 2. 多维度组合查询（最多 3 个维度）
```
# 用户 + 终端类型
GET /sessions?user_id=12345&device_type=mobile

# IP 段 + 会话类型
GET /sessions?ip_prefix=192.168.1.0/24&session_type=web

# 用户 + IP + 终端类型
GET /sessions?user_id=12345&ip=192.168.1.100&device_type=mobile
```

#### 3. 范围查询
```
# 创建时间范围
GET /sessions?created_after=2025-12-05T00:00:00Z&created_before=2025-12-05T23:59:59Z

# 最后活跃时间范围
GET /sessions?last_active_before=2025-12-05T12:00:00Z
```

#### 4. 聚合统计
```
# 按终端类型统计在线用户数
GET /stats/sessions/count?group_by=device_type

# 按令牌类型统计分布
GET /stats/tokens/count?group_by=token_type

# 按 IP 段统计会话数
GET /stats/sessions/count?group_by=ip_prefix
```

### 查询性能保证

- **简单查询（单维度）**：P99 < 1ms
- **复杂查询（多维度）**：P99 < 5ms
- **聚合统计**：P99 < 10ms（可配置缓存）

## 补充说明

### 与 Redis 的对比

| 能力 | Redis | TokMesh |
|------|-------|---------|
| 多维度查询 | ❌ 需复杂 key 设计 | ✅ 原生支持 6 大维度 |
| TTL 策略 | ⚠️ 简单 EXPIRE | ✅ 3 种模型 + 策略化配置 |
| 过期清理 | ⚠️ 可能有抖动 | ✅ 渐进式清理 |
| 查询性能 | ✅ 极快（单 key） | ✅ 快（多维度优化） |
| 数据建模 | ⚠️ KV 存储，需手动建模 | ✅ 领域专用模型 |

### 典型使用场景

1. **安全运营**：
   - 查询来自异常 IP 段的所有会话
   - 检测单用户在多设备的异常登录
   - 统计管理令牌的使用频率

2. **用户体验优化**：
   - Mobile 端使用 Sliding TTL，保持用户登录状态
   - Web 端使用固定 TTL + 刷新令牌，平衡安全与体验

3. **合规审计**：
   - 查询某时间段内的所有会话创建记录
   - 按用户类型统计会话分布
   - 导出特定 IP 段的访问记录
