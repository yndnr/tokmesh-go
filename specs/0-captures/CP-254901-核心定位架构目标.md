# 想法: TokMesh 核心定位、架构目标与设计原则

状态: 草稿
创建日期: 2025-12-05
编号: CP-254901

## 原始想法

构建一个**专为 SSO/IAM/OAuth/OIDC 设计的高性能会话/令牌缓存与校验加速器**，定位为"校验加速器"而非通用缓存，让 SSO/IAM 专注于认证逻辑，由 TokMesh 承担高并发校验压力。

## 核心定位

### 专用领域

本服务专为以下场景的"会话 / 令牌管理与校验"而设计：

- **SSO / 统一身份认证中心**
- **IAM 系统**
- **OAuth / OIDC 提供方**
- **内部统一登录平台与各类业务系统网关**

### 与现有方案（Redis）的差异

**不是什么**：
- ❌ 不是通用 KV 缓存（不试图替代 Redis 的所有用途）
- ❌ 不负责前端渲染或浏览器安全防护的具体实现
- ❌ 不做复杂数据分析（应由专用数据仓库承担）

**是什么**：
- ✅ 会话 / 令牌场景的**领域专用存储**
- ✅ SSO/IAM 的**高性能校验加速器**
- ✅ 在保证安全的前提下追求**极致性能**

### 价值主张

让 SSO 可以对业务方宣称：
> "基础认证仍由 SSO 完成，但在线会话 / 令牌的高速校验由 TokMesh 承担，性能和并发能力远优于直接由 SSO 自身承担校验压力。"

## 性能目标

### 极致性能指标

- **QPS 目标**: 10万+ QPS（单节点或集群）
- **延迟目标**: P99 延迟 < 1ms
- **目标规模**: 十万级在线用户 + 支持弹性扩展

### 性能优先的权衡

- 以内存作为高速缓存介质
- 允许在极端情况下牺牲少量会话一致性（例如少量数据丢失导致用户需重新登录），以换取整体性能与可用性
- 但在**安全性上不妥协**：采用安全优先策略，所有数据强制加密

## 核心目标

1. **高性能的会话 / 令牌存取与校验**
   - 专用数据结构优化
   - 低延迟访问路径

2. **多维度但受控复杂度的检索与统计**
   - 支持 6 大维度检索（用户、IP、会话、令牌、终端、类型）
   - 拒绝无边界复杂查询，确保性能可预测

3. **可控、可审计的在线下线 / 踢人治理能力**
   - 管理 API 与业务 API 严格分离
   - 完整的审计日志

4. **易于从现有 Redis 方案迁移**
   - 支持 Redis 协议兼容
   - 提供 SDK / 客户端库
   - 提供 RESTful API

5. **支持分布式集群与数据分片**
   - 单节点到分布式集群的渐进式部署
   - Raft/Paxos 一致性协议
   - 水平扩展能力

6. **具备内存加密、落盘加密等安全能力**
   - 全链路加密（内存 + 落盘 + 传输）
   - 敏感字段脱敏
   - 密钥管理与轮换

7. **支持 IAM / 安全运营平台通过管理接口进行统一管理**
   - 会话管理（查看、搜索、踢出）
   - 配置管理（动态配置、限流策略）
   - 监控告警（实时指标、健康检查）

## 设计约束与协作原则

### 核心约束

1. **性能优先，但拒绝无边界复杂查询**
   - 查询能力受控，确保性能可预测
   - 复杂分析查询应由专用数据仓库承担

2. **坚持领域专用存储，而非通用缓存**
   - 专注于会话/令牌场景
   - 不试图成为通用 KV 存储

3. **严格区分业务校验接口与管理接口**
   - **业务 API**：极致性能，简单校验，高 QPS
   - **管理 API**：完整审计，严格限流，安全优先

4. **Web 安全职责边界**
   - XSS/CSRF 由上层 Web 应用负责具体实现
   - TokMesh 提供必要支撑字段和约束
   - 不直接承担浏览器防护逻辑

### 冲突处理原则

若新需求与性能、安全、领域边界等原则冲突：
1. 在设计中明确指出冲突点
2. 给出替代方案
3. 由架构委员会评审决策

### 依赖策略

- **外部依赖尽可能最小**：常用库倾向内置或静态链接
- **大型依赖评估**：若仅使用少量功能，优先考虑替代或自研精简实现
- **Go 语言优势**：单一二进制文件，无运行时依赖

## 架构愿景

### 渐进式部署模式

- **阶段 1**: 单节点模式（小规模场景或测试环境）
- **阶段 2**: 主从复制模式（提高读性能和可用性）
- **阶段 3**: 分布式集群模式（数据分片 + 多节点，水平扩展）
- **阶段 4**: Raft/Paxos 一致性（强一致性，数据不丢失）

### 数据持久化策略

- **AOF/WAL 日志**：追加写日志，支持故障恢复
- **分级存储**：热数据在内存，冷数据自动下沉到磁盘

## 补充说明

### 目标用户

- **主要用户**：企业级 SSO/IAM 团队、安全运营团队
- **典型场景**：
  - 大型互联网公司的核心认证系统
  - 中大型企业的统一身份认证平台
  - 金融、政务等高安全场景的 IAM 系统

### 与上下游系统的协作关系

- **上游（SSO/IAM）**：负责用户认证、令牌签发、策略决策
- **本项目（TokMesh）**：负责高性能会话/令牌缓存、快速校验、多维度检索
- **下游（业务系统）**：负责业务逻辑，调用 TokMesh 进行令牌校验

### 非目标（明确不做什么）

- ❌ **不负责用户认证逻辑**：认证由 SSO/IAM 完成
- ❌ **不负责令牌签发与加密**：令牌生成由 SSO/IAM 完成
- ❌ **不负责策略决策**：授权策略由 IAM 完成
- ❌ **不做复杂数据分析**：OLAP 查询应由数据仓库承担
- ❌ **不做浏览器前端防护**：XSS/CSRF 具体实现由 Web 应用承担

### 关键成功因素

- ✅ 在保证安全的前提下实现极致性能（10万+ QPS，P99 < 1ms）
- ✅ 多维度检索能力（相比 Redis 的核心优势）
- ✅ 易于迁移（Redis 协议兼容）
- ✅ 企业级部署支持（跨平台、容器化、K8s 友好）
- ✅ 完整的可观测性（Prometheus 原生集成）
