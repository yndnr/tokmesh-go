# 想法: 企业级部署能力

状态: 草稿
创建日期: 2025-12-05
编号: CP-254904
关联文档: CP-254901

## 原始想法

TokMesh 支持**跨平台部署 + 容器化 + Kubernetes 原生 + 多种 TLS 模式 + 系统加固**，适配各种企业级基础设施环境（云、IDC、混合云），满足不同安全策略要求。

## 跨平台支持

### 支持的操作系统与架构

#### 操作系统
- **Windows**：Windows Server 2019+, Windows 10+
- **Linux**：主流发行版（Ubuntu、Debian、CentOS、RHEL、Rocky Linux）
- **macOS**（可选）：开发和测试环境

#### 架构
- **x86_64 / AMD64**：主流服务器架构
- **arm64 / aarch64**：ARM 服务器、云原生场景

### 平台特定支持

#### Windows 平台

**Windows Service 集成**：
- 支持 Windows Service 安装和管理
- 使用 `sc create` / `nssm` 等工具注册服务
- 支持自动启动、重启策略

**受限 Service Account**：
- **非管理员权限运行**
- 使用受限 Service Account（如 `NT SERVICE\TokMesh`）
- 最小权限原则

**Windows 事件日志集成**：
- 集成 Windows Event Log
- 支持事件查看器（Event Viewer）查看日志

#### Linux 平台

**systemd 集成**：
- 提供 systemd service 文件（`tokmesh.service`）
- 支持 `systemctl start/stop/restart/status`
- 支持自动重启策略（`Restart=on-failure`）

**非 root 账户运行**：
- **最小权限原则**，使用专用账户（如 `tokmesh` 用户）
- 避免使用 root 权限运行
- 示例：
  ```ini
  [Service]
  User=tokmesh
  Group=tokmesh
  ```

**SELinux / AppArmor 加固**：
- 支持 SELinux 策略（RHEL / CentOS）
- 支持 AppArmor 配置（Ubuntu / Debian）
- 提供默认安全策略文件

**Linux Capabilities**：
- 使用 Linux Capabilities 精细化权限控制
- 仅授予必要的 capabilities（如 `CAP_NET_BIND_SERVICE`）
- 示例：
  ```ini
  [Service]
  AmbientCapabilities=CAP_NET_BIND_SERVICE
  ```

## 容器化与 Kubernetes

### Docker / Podman 支持

#### 官方镜像

**多架构支持**：
- 提供 `amd64` 和 `arm64` 镜像
- 使用 Docker Buildx 或 Podman 构建多架构镜像
- 镜像仓库：Docker Hub / GitHub Container Registry / 私有仓库

**非 root 用户运行**：
- Dockerfile 中使用 `USER` 指令
- UID > 1000（如 `USER 10001`）
- 避免容器内使用 root 权限

**最小化镜像**：
- 基于 **Alpine Linux**（轻量，约 10-20 MB）
- 或基于 **Distroless**（无 shell，更安全）
- 仅包含必要的二进制文件和依赖

#### 镜像示例

```dockerfile
FROM golang:1.21-alpine AS builder
WORKDIR /app
COPY . .
RUN go build -o tokmesh ./cmd/tokmesh

FROM alpine:3.18
RUN addgroup -g 10001 tokmesh && adduser -D -u 10001 -G tokmesh tokmesh
USER 10001
COPY --from=builder /app/tokmesh /usr/local/bin/tokmesh
EXPOSE 8080 9090
ENTRYPOINT ["/usr/local/bin/tokmesh"]
```

### Kubernetes 原生支持

#### Health Probes（健康检查）

**Liveness Probe**（存活探针）：
- 检测服务是否存活
- 失败时 Kubernetes 重启 Pod
- 示例：`GET /health/live`

**Readiness Probe**（就绪探针）：
- 检测服务是否准备好接收流量
- 失败时从 Service 中移除 Pod
- 示例：`GET /health/ready`

**Startup Probe**（启动探针）：
- 检测服务是否启动完成
- 适用于启动较慢的服务
- 示例：`GET /health/startup`

#### HPA（Horizontal Pod Autoscaler）

**水平扩缩容支持**：
- 基于 CPU / 内存使用率自动扩缩容
- 基于自定义指标（如 QPS、延迟）扩缩容
- 示例：
  ```yaml
  apiVersion: autoscaling/v2
  kind: HorizontalPodAutoscaler
  metadata:
    name: tokmesh-hpa
  spec:
    scaleTargetRef:
      apiVersion: apps/v1
      kind: Deployment
      name: tokmesh
    minReplicas: 2
    maxReplicas: 10
    metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
  ```

#### ConfigMap / Secret 集成

**ConfigMap**（配置管理）：
- 将配置文件挂载为 ConfigMap
- 支持配置热重载（监听文件变化）

**Secret**（密钥管理）：
- 敏感信息（密钥、证书）存储在 Secret
- 挂载为文件或环境变量

**示例**：
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: tokmesh
spec:
  containers:
  - name: tokmesh
    image: tokmesh:latest
    volumeMounts:
    - name: config
      mountPath: /etc/tokmesh
    - name: tls-certs
      mountPath: /etc/tls
  volumes:
  - name: config
    configMap:
      name: tokmesh-config
  - name: tls-certs
    secret:
      secretName: tokmesh-tls
```

#### StatefulSet（有状态部署）

**支持有状态部署**：
- 分布式集群模式下使用 StatefulSet
- 保证 Pod 有稳定的网络标识（如 `tokmesh-0`, `tokmesh-1`）
- 支持持久化存储（PersistentVolumeClaim）

#### PodDisruptionBudget（优雅滚动更新）

**最小可用 Pod 数**：
- 保证滚动更新时至少 N 个 Pod 可用
- 避免全部 Pod 同时重启
- 示例：
  ```yaml
  apiVersion: policy/v1
  kind: PodDisruptionBudget
  metadata:
    name: tokmesh-pdb
  spec:
    minAvailable: 2
    selector:
      matchLabels:
        app: tokmesh
  ```

## 负载均衡与 TLS 终止模式

### API 分层设计

**业务 API**：
- 高性能令牌/会话校验接口
- 面向业务系统
- 端口：如 `8080`

**管理 API**：
- 会话管理、配置、审计接口
- 面向运维/安全团队
- 端口：如 `9090`

**独立部署**：
- 可分别置于不同负载均衡之后
- 实现隔离和差异化策略

### 支持的 TLS 终止模式

#### 1. SSL Offload（LB 终止 SSL）

**场景**：内网环境，性能最优

```
客户端 --[HTTPS]--> LB --[HTTP]--> TokMesh
```

**优点**：
- LB 处理 TLS 握手和解密
- TokMesh 无需处理 TLS，性能最优
- 适合内网环境，LB 到后端的链路可信

**缺点**：
- LB 到后端使用明文通信
- 不适合跨网络边界的部署

#### 2. Re-encrypt（LB 终止 + 后端再加密）

**场景**：安全合规要求较高

```
客户端 --[HTTPS]--> LB --[HTTPS]--> TokMesh
```

**优点**：
- LB 解密后再加密传输到后端
- 后端再次终止 TLS
- 适合安全合规场景

**缺点**：
- 双重 TLS 握手，性能略低
- 配置复杂度较高

#### 3. Passthrough（直透，后端终止 TLS）

**场景**：端到端加密，安全性最高

```
客户端 --[HTTPS]--> LB --[TCP]--> TokMesh (终止 TLS)
```

**优点**：
- LB 仅做 L4 转发，不处理 TLS
- 后端直接终止 TLS
- 端到端加密，安全性最高

**缺点**：
- LB 无法基于 HTTP 内容做路由
- 后端需要处理 TLS，性能略低

#### 4. mTLS 模式（双向 TLS 认证）

**场景**：管理 API，客户端身份验证

```
客户端 (有证书) --[mTLS]--> LB --[mTLS]--> TokMesh
```

**优点**：
- 客户端证书身份绑定
- 强认证，防止未授权访问
- 推荐用于管理 API

**配置示例**：
```yaml
tls:
  mode: mtls
  cert_file: /etc/tls/server.crt
  key_file: /etc/tls/server.key
  ca_file: /etc/tls/ca.crt  # 用于验证客户端证书
  client_auth: require      # 要求客户端证书
```

### 负载均衡支持

#### L4 负载均衡（基于 TCP/IP）
- 性能最优
- 适合 Passthrough 模式
- 示例：HAProxy、Nginx Stream、云 LB

#### L7 负载均衡（基于 HTTP/gRPC）
- 支持更丰富的路由策略
- 可基于 URL、Header 路由
- 示例：Nginx、Envoy、云 LB

## 依赖管理策略

### 最小化外部依赖

**原则**：
- 常用库倾向**内置或静态链接**
- 减少运行时依赖
- 避免动态链接库版本冲突

**优势**：
- 提供**独立二进制文件**（standalone binary）
- 无需安装额外依赖
- 简化部署和分发

### 大型依赖评估原则

若仅使用少量功能，优先考虑：
1. **寻找更轻量的替代库**
2. **自研精简实现**
3. **评估是否真正必要**

**示例**：
- ❌ 避免：为了一个功能引入整个重量级框架
- ✅ 优选：使用轻量库或自研精简实现

### Go 语言优势

**单一二进制文件**：
- 无运行时依赖（无需安装 Go 环境）
- 交叉编译支持多平台
- 内存安全，无需 C 运行时

**编译示例**：
```bash
# Linux amd64
GOOS=linux GOARCH=amd64 go build -o tokmesh-linux-amd64

# Linux arm64
GOOS=linux GOARCH=arm64 go build -o tokmesh-linux-arm64

# Windows amd64
GOOS=windows GOARCH=amd64 go build -o tokmesh-windows-amd64.exe
```

## 补充说明

### 部署模式总结

| 部署模式 | 适用场景 | 优势 |
|---------|---------|------|
| 单节点 | 小规模、测试环境 | 简单、快速部署 |
| Docker | 开发、测试、生产 | 隔离、可移植 |
| Kubernetes | 大规模生产环境 | 弹性、高可用、易管理 |
| systemd | 传统 Linux 环境 | 稳定、易管理 |
| Windows Service | Windows 环境 | 原生集成 |

### 安全加固总结

**系统级加固**：
- 非 root / 非管理员运行
- SELinux / AppArmor 策略
- Linux Capabilities 精细化权限

**容器级加固**：
- 非 root 用户（UID > 1000）
- 最小化镜像（Alpine / Distroless）
- 只读文件系统（可选）

**网络级加固**：
- 多种 TLS 模式（SSL Offload / Re-encrypt / Passthrough / mTLS）
- API 分层隔离
- 防火墙策略
