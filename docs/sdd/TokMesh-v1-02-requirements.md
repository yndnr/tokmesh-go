# TokMesh v1 需求清单（Requirements）

> **目标**：用一份干净的列表，收敛 TokMesh v1 的核心需求。每条需求都有简单编号（R1, R2, ...）、简短说明与关键约束，方便后续设计与实现对齐。

## 约定

- 编号 R1, R2, ... 仅用于引用和沟通，不强制与实现文件结构一一对应。  
- 若某条需求后续需要详细展开，可以在设计文档中引用其编号。  
- 新增需求时，优先判断是否是对现有 R 的补充，避免重复。

### 实施阶段说明（P1–P4）

- 为便于规划实现顺序，本文件为每条需求标注一个**实施阶段**：`P1`、`P2`、`P3`、`P4`。含义为：  
  - **P1：MVP + mTLS**  
    - 单节点或简单拓扑即可满足的核心能力：会话模型、多维索引、生命周期 API、基础持久化与资源管理、基础 PKI/mTLS、安全的端口划分、最小可用的协议与部署形态。  
  - **P2：加密与安全增强（内存 + 落盘 + 端到端 + 主动防御 + App 校验）**  
    - 在 P1 的基础上补强会话/令牌的端到端加密存储能力，完善主动防御、证书轮换增强、App 直连校验、配置体验与令牌类型策略。  
  - **P3：集群化与一致性**  
    - 引入真正的去中心化集群、一致哈希分片、副本与 Any-Node Gateway 等能力，支撑多节点高可用和扩展性。  
  - **P4：SDK 与生态**  
    - 完成多语言 SDK 与更加友好的接入体验，将 TokMesh 从“可用的基础设施”演进为“易用的产品生态”。  
- 实施阶段是规划性的，可随项目进展调整，但应通过更新本文件保持**单一真相源**。

## 核心功能类

### R1. TokMesh 专用会话存储

- **类型**：功能 / 定位  
- **描述**：  
  - TokMesh 是“面向身份认证（IAM/SSO）的 Session/Token 存储系统”，不是通用 KV 数据库。  
  - 主要目标是替代 Redis 在 SSO 场景中的会话存储角色。  
  - TokMesh 对上层使用的具体 SSO 协议（如 OAuth2/OIDC、SAML、CAS 等）保持协议无关，仅管理这些协议在完成认证之后产生的会话 / 令牌状态。  
- **关键约束 / 建议**：  
  - 不追求 Redis 全量命令兼容，只支持 SSO 必需的子集。  
  - 所有后续需求不应把 TokMesh 推进成“通用数据库”。
  - **实施阶段**：P1（MVP 核心定位）

### R2. 多维会话索引与 O(1) 踢人

- **类型**：功能  
- **描述**：  
  - 会话支持按 User、Client/Device、App、Tenant/环境 等维度查询。  
  - 管理侧可基于这些维度对会话进行查询、统计与批量撤销/踢人：  
    - **单节点场景**：  
      - 按 SessionID 踢人可视为 O(1)；  
      - 按 User/Device/Tenant 等维度批量踢人，其复杂度与该维度下会话数量 k 成正比（O(k)），但无需扫描全量会话；  
    - **集群场景（P3 以后）**：  
      - 按 SessionID 踢人通过一致哈希或等效路由可保持近似 O(1)；  
      - 按 User/Device/Tenant 维度的批量踢人在实现上为 scatter-gather 操作，复杂度与涉及节点数 N 和各节点上相关会话数 k_i 有关，属于“近似 O(N_nodes + Σk_i)” 能力，而非严格 O(1)。  
- **关键约束 / 建议**：  
  - 多维索引是“一等公民”，不是附加 feature。  
  - 设计时需关注索引一致性和更新成本。
  - 对外暴露的索引字段（如 user_id / device_id / tenant / app_id 等）在 API 层应有明确长度上限（建议：user_id/device_id/app_id ≤ 128 字节，tenant ≤ 64 字节），避免超长输入导致内存与 CPU 被滥用；内部索引实现可以对多维组合键进行哈希压缩为固定长度 key，但必须保留原始字段，并通过桶内比对确保按 tenant/user/device 等维度的精确查询、统计与踢人能力不受影响（不要求前缀/范围查询）。
  - **实施阶段**：P1（MVP 核心能力）

### R3. 会话全生命周期管理

- **类型**：功能  
- **描述**：  
  - 支持会话的完整生命周期：写入、查询、原子续期（Replay-Extension）、撤销/踢人、TTL 过期清理。  
  - 为审计与安全分析预留必要数据（如操作记录、状态变化点）。  
- **关键约束 / 建议**：  
  - 生命周期语义要严谨，避免并发续期、过期边界问题。  
  - 审计字段和日志量要可控，避免 I/O 风暴。
  - 时间语义需可预期：  
    - 所有与 TTL、创建时间、最后活动时间相关的判断统一基于 UTC 时间；  
    - 在集群模式下，生产环境必须配置 NTP 或等效时间同步机制，并约定最大允许 Clock Skew（建议 ≤ 60 秒，可配置）；  
    - 在 TTL 判断等边界场景可预留小幅容忍窗口，其绝对值应不大于最大 Clock Skew 的一半（例如 Clock Skew ≤ 60 秒时，容忍窗口 ≤ ±30 秒，具体数值在 SDD-2 中细化），以避免不同节点间对同一会话/令牌有效性的判断出现过大的偏差。
  - **实施阶段**：P1（MVP 核心能力）

### R4. 会话生命周期 API（内部给 SSO/IAM 使用）

- **类型**：接口 / 功能  
- **归属平面**：业务数据平面（内部）  
- **描述**：  
  - TokMesh 提供给 SSO/IAM 侧的内部 API：  
    - `CreateSession` / `ValidateToken` / `ExtendSession` / `RevokeSession` 等。  
  - 这些 API 是会话行为的“权威入口”。  
- **关键约束 / 建议**：  
  - App 不直接调用写入/续期类接口（见 R10）。  
  - API 行为必须与 R1–R3 定义的模型和生命周期严格一致。
  - **实施阶段**：P1（MVP 核心接口）

## 安全 & 零信任类

### R5. 零依赖 PKI（基于 Go 标准库）

- **类型**：安全  
- **描述**：  
  - 所有证书与加密逻辑使用 Go 标准库实现，不依赖 OpenSSL 等外部库。  
  - TokMesh 内置基本 PKI 能力，支持生成或导入 CA / 服务端 / 客户端证书。  
- **关键约束 / 建议**：  
  - 支持导入企业 PKI 的 CA/证书作为可选能力。  
  - 未来如对接 HSM/KMS，需在此基础上扩展，而不是推翻。
  - TokMesh 应提供基于 `tokmesh-cli` 的证书运维能力，包括：  
    - 生成内部 CA、服务端证书、客户端证书（供 SSO/IAM 服务和 `tokmesh-cli` 使用）；  
    - 导入外部证书和 CA（例如 LB 设备签发的 HTTPS 证书或企业 PKI 证书）；  
    - 查看当前部署中各节点使用的证书及其到期时间。  
  - 证书存放与格式约定：  
    - 默认使用 PEM 编码的证书与私钥文件（如 `ca.pem`、`server.pem`、`server-key.pem`），存放于可配置的证书目录（例如 Linux 下 `/etc/tokmesh/certs/`，Windows 下 `%PROGRAMDATA%\\TokMesh\\certs\\`）；  
    - 私钥文件权限应严格受控（如仅 TokMesh 进程用户可读），并在文档中提供最低权限建议。
  - **实施阶段**：P1（MVP 基础 PKI；高级集成可在后续阶段增强）

### R6. 全链路 mTLS 与零信任前提

- **类型**：安全  
- **描述**：  
  - 集群内部、管理 API、内部服务访问（如 SSO→TokMesh）应支持，并在生产环境推荐强制 mTLS。  
  - TokMesh 假定整体网络“不可信”，不因“内网”而放宽安全要求。  
- **关键约束 / 建议**：  
  - 明确哪些端口/接口必须 mTLS，哪些可以是单向 TLS（如 App 侧，只读 API，见 R10）。  
  - 与 R5 中的 PKI 能力保持一致的信任模型。
  - 对通过官方 SDK 接入的 SSO/IAM 服务，在生产环境中应默认使用 mTLS（客户端证书可由 TokMesh 内置 CA 或企业 PKI 签发），仅在开发/测试环境可选择降级为单向 TLS。
  - 对管理控制平面的访问（如 `tokmesh-cli`、IAM 系统集成的管理后台）在生产环境中必须使用 mTLS，以确保高权限操作来源可被强身份识别和追踪。
  - **实施阶段**：P1（MVP 必需的安全基线；业务平面可按场景渐进）

### R7. 内存数据加密

- **类型**：安全  
- **描述**：  
  - 内存中存放的敏感 Session/Token 数据应以加密或混淆方式保存，降低节点被入侵或内存 dump 时的泄露风险。  
- **关键约束 / 建议**：  
  - 加密算法选型：优先使用标准库提供的 AES-GCM 系列，支持 `AES-256-GCM`（默认）与 `AES-128-GCM` 两种模式，可通过配置选择；  
  - 运行时应优先利用 CPU 的 AES 指令集（如 AES-NI），在缺失硬件加速的环境下可回退到纯软件实现，并在文档中说明性能差异；  
  - 需要评估加密带来的 CPU 和延迟开销，并在 SDD-2/实现阶段通过 benchmark 量化。  
  - 密钥管理、刷新策略要简洁可控。  
  - 内存加密的威胁模型应与落盘加密（见 R17）保持一致：聚焦于防御获得进程内存或本地存储访问能力的攻击者，不过度承诺“绝对安全”。
  - **实施阶段**：P2（加密增强：内存加密）

### R8. 主动防御：重放防御 + 黑白名单 + 豁免 IP

- **类型**：安全  
- **描述**：  
  - TokMesh 在协议和应用层内建主动防御能力：  
    - 防重放攻击。  
    - 自动黑名单：对恶意 IP/Client 进行自动封禁。  
    - 白名单与豁免 IP：允许对监控/管理等受信来源做特殊放行。  
- **关键约束 / 建议**：  
  - 防御策略的阈值（例如失败次数、请求频率）应可配置。  
  - 需避免误伤严重影响正常业务，提供撤销/解封机制。  
  - 对令牌校验/只读校验 API，应内建按来源 IP 的基础防护能力：  
    - 支持按 IP 的速率限制（如每秒/每分钟最大校验次数），用于保护后端资源不被单一来源拖垮；  
    - 支持按 IP 的错误率熔断（例如在短时间窗口内 401/403 比例过高时，临时冻结该 IP 一段时间）；  
    - 上述阈值和冻结时长均需要通过配置控制，默认值偏保守，同时应通过日志与指标（参见 R22）暴露防护触发情况，便于运维调优。
  - **实施阶段**：P2（加密与安全增强阶段，可在 P1 基础上逐步增强主动防御）

### R9. 证书与通讯密钥轮换（集群级）

- **类型**：安全 / 运维  
- **描述**：  
  - TokMesh 支持对证书与集群内部通讯密钥进行周期性的轮换：  
    - 证书有明确到期时间，系统可检测并告警。  
    - 通过 `tokmesh-cli` 对单节点或整个集群进行证书更新。  
    - 集群内部通讯密钥/密码具备轮换机制，支持周期或手动触发。  
- **关键约束 / 建议**：  
  - 轮换过程应尽量“无感”，保持业务连续性。  
  - 需考虑 LB、代理等组件的证书链一致性与协同。
  - `tokmesh-cli` 应支持在集群范围内进行证书分发与更新：  
    - 支持对单节点或一组节点下发新的证书和密钥（基于管理控制平面的 API）。  
    - 支持滚动更新模式，避免全体节点同时切换导致中断。  
  - 每个 TokMesh 节点应有明确的“证书更新接收机制”：  
    - 可以从本地安全存储（文件系统或操作系统证书存储）中加载新证书；  
    - 在权限受限环境下，应定义最小必要权限（如证书文件的路径与权限要求），并在文档中说明；  
    - 更新过程中的旧证书保留策略、回滚策略需有基本约定，以便在证书更新失败时恢复。  
  - 节点证书版本与回滚：  
    - 每个 TokMesh 节点应支持维护多版本证书文件，并通过“当前活动版本”的指针（如配置项或软链接）选择实际生效的证书；  
    - 证书轮换时，先将新证书写入为新版本，再更新活动指针并热加载；  
    - 发生问题时，应支持通过管理接口将活动指针指回上一版本，实现快速回滚；  
    - 支持外部配置管理系统（如 Ansible、K8S Secret 等）负责写入证书文件，TokMesh 只负责读取与切换。  
  - 通讯密钥 / 密码轮换原则：  
    - 集群节点间通讯加密应以 TLS/mTLS 为主，具体会话密钥由 TLS 协议自动协商和定期更新，TokMesh 无需额外实现自定义的会话密钥轮换协议；  
    - 证书轮换本身就会触发新的会话密钥协商，是节点间通讯长期安全性的主要保障手段；  
    - 如未来引入额外的“集群级对称密钥”（例如对应用层载荷再加一层加密），则该密钥也应具备版本号/epoch 的概念：  
      - tokmesh-cli 分发新密钥至各节点；  
      - 节点在一段过渡期内同时接受旧/新密钥（读双写单），并在切换完成后淘汰旧密钥；  
      - 相关流程与证书轮换类似，应提供基本的回滚机制。
  - **实施阶段**：P2（加密与安全增强阶段；v1.0 可先具备基本更新能力）

### R23. 审计日志与安全事件记录

- **类型**：安全 / 合规 / 可观测性  
- **描述**：  
  - TokMesh 需要对关键安全事件和高风险操作进行结构化审计记录，以支撑安全分析、合规审查和事后追踪。  
- **关键约束 / 建议**：  
  - 审计范围：至少覆盖以下操作与事件：  
    - 会话/令牌层面的敏感操作（创建、批量撤销/踢人、异常失败）；  
    - 管理操作（配置变更、证书与密钥轮换、黑白名单/豁免列表变更）；  
    - 主动防御触发（见 R8）、异常访问模式（如高错误率来源 IP/Client）。  
  - 日志内容与格式：  
    - 使用结构化格式（如 JSON），至少包含时间戳（UTC）、节点标识、操作者身份（如 Client_ID 或管理主体）、操作类型、目标范围以及结果状态；  
    - 对 Token 内容等高敏感字段应进行脱敏或散列处理，避免在日志中直接暴露明文；  
    - 在需求或后续 SDD-2 中需明确敏感字段分级示例：  
      - 高敏感：Token Payload、Refresh Token、API Key、密码摘要等绝不明文记录；  
      - 中敏感：SessionID、UserID、IP 地址等可按策略做部分脱敏（如截断或掩码）；  
      - 低敏感：TenantID、简单标签等可明文记录。  
    - 审计日志应支持输出到本地文件，并可通过日志系统（如 ELK）集中采集。  
  - 保留与防篡改：  
    - 提供日志保留期相关配置（如保留最近 N 天或按容量上限轮转）；  
    - 在文档中给出防篡改建议（如只追加文件、只读挂载、外部审计存储等），TokMesh 本身不强制实现完整的防篡改存储，但需要为集成留出接口与结构化信息。  
  - 加密与关联性：  
    - 审计日志文件中如包含敏感字段（即便已脱敏），在磁盘上应视为敏感持久化内容，与 R17 对齐，可选支持对审计日志文件进行加密或加密其中的高敏感字段；  
    - 审计日志结构应预留 `request_id`/`trace_id`、`endpoint`、`result` 等字段，便于与 R22 暴露的指标、未来的分布式追踪信息进行关联。  
- **验收视角 / 测试要点**：  
  - 在典型管理与安全事件（批量踢人、黑名单触发、证书轮换）下，能生成包含完整上下文的审计日志，并可通过简单查询恢复事件序列。  
  - 在大流量场景下，审计日志写入不会成为明显瓶颈，必要时可通过异步写入或采样机制控制开销；在后续 SDD-2/实现阶段应对日志写入延迟与对核心请求路径（如 ValidateToken）的性能影响给出可测的 benchmark 目标。  
  - **实施阶段**：P2（在基础功能与安全能力之上补齐审计与合规支撑）

## App 直连类

### R10. App 直连 TokMesh 的只读校验 API

- **类型**：功能 / 安全 / 接入  
- **归属平面**：业务数据平面（外部只读）  
- **描述**：  
  - 提供一组供业务 App 直接调用的只读校验 API，用于验证 Token/Session 是否有效（如 `ValidateToken` / `GetSessionStatus`）。  
  - 主要目标是减轻 SSO 服务的校验压力。  
- **关键约束 / 建议**：  
  - **只读原则**：  
    - App 直连通道只允许校验和只读查询，不允许写入/续期/撤销等操作。  
    - 写入/续期/撤销必须由 SSO/IAM 服务通过内部 API 完成（R4）。  
  - **传输层安全**：  
    - 默认对外暴露的校验端口应强制开启 TLS（HTTPS），即使单节点部署亦然。  
    - 在单节点场景，可直接使用 IAM 管理员提供的服务端证书，实现 HTTPS 访问。  
    - 在有 LB 的场景：  
      - App→LB：使用 LB 签发或托管的证书做 HTTPS 终止。  
      - LB→TokMesh：  
        - 可以继续使用 TLS，由 TokMesh 信任 LB 所用的 CA 证书；  
        - 或者在安全要求较高时启用 mTLS，但不是强制。  
  - **mTLS 使用范围**：  
    - mTLS 主要面向：  
      - IAM/SSO 后端服务接入 TokMesh；  
      - TokMesh 集群内部节点间通信；  
      - `tokmesh-cli` 等管理客户端。  
    - App 侧直连只读 API 不要求 mTLS（否则接入门槛过高），但必须强制 HTTPS + 应用级鉴权（API Key / Client ID / 签名等）。  
  - **暴露方式**：  
    - 校验 API 对 App（含浏览器客户端）暴露时，可以有多种部署选择：  
      - 直接暴露一个 TokMesh 节点作为 HTTPS 入口；  
      - 暴露多个节点地址，由客户端通过 DNS 轮询进行访问；  
      - 通过 LB 设备统一暴露一个入口，由 LB 根据策略转发到后端节点。  
    - 推荐由管理员统一配置 LB 或 DNS 轮询来暴露 TokMesh 校验服务，以便：  
      - 统一做接入控制（TLS 终止、访问控制、限流、防御策略）。  
      - 方便扩容、灰度升级与节点下线。  
    - 无论 App 通过哪个节点访问校验 API，TokMesh 集群都应保证校验结果语义一致。  
  - **协议无关性**：  
    - 校验 API 仅提供与会话/令牌状态相关的通用查询能力，不暴露任何特定协议（如 OAuth2/OIDC、SAML、CAS）的流程端点。  
    - 上层 SSO 协议负责完成登录 / 授权流程并生成令牌，TokMesh 只负责管理和校验这些令牌的状态。
  - **防枚举与限流配合**：  
    - App 直连校验场景应默认启用针对校验端点的速率限制与错误率熔断机制，可由 TokMesh 内建（见 R8）并与 LB/WAF 的策略协同；  
    - 对接入方文档中需要明确说明：大规模 Token 扫描/校验行为会触发防护，接入方应通过 Client_ID / API_Key / 签名等方式进行鉴权与配额管理，避免将业务误用为批量校验/探测通道。
  - **实施阶段**：P2（在 P1 的内部调用基础上，对 App 开放只读校验能力）

## 可靠性 & 集群一致性 & 资源类

### R11. 会话持久化与快速恢复

- **类型**：可靠性  
- **描述**：  
  - TokMesh 支持将会话数据落盘（单节点/集群节点）。  
  - 在节点或集群重启时，能够从持久化数据中快速恢复关键会话，尽量避免大量用户重新登录。  
- **关键约束 / 建议**：  
  - 落盘策略（如 WAL + 快照）与恢复策略需配合控制 RTO/RPO。  
  - 恢复过程中不能长时间“冻住”服务或造成极端性能抖动。
  - **实施阶段**：P1（MVP 持久化能力）

### R12. 内存阈值与节点资源管理

- **类型**：可靠性 / 资源  
- **描述**：  
  - 节点支持配置最大内存使用阈值。  
  - 当内存使用接近或超过阈值时，TokMesh 能采取清晰且可配置的策略（如淘汰、只读、限流），并发出告警。  
- **关键约束 / 建议**：  
  - 默认策略应偏向“保护稳定性”，宁可拒绝写入，不允许 OOM 崩溃。  
  - 驱逐策略需兼顾业务价值（优先淘汰低价值会话）。
  - **实施阶段**：P1（MVP 资源保护能力）

### R24. 数据保留与清理策略（内存 + 持久化）

- **类型**：可靠性 / 运维 / 合规  
- **描述**：  
  - TokMesh 需要在内存与持久化层面定义清晰的数据保留与清理策略，以控制资源占用并满足合规要求。  
- **关键约束 / 建议**：  
  - 过期数据清理：  
    - Session/Token 在 TTL 到期后应从内存与持久化介质中被清理，清理策略可以结合定期后台任务与懒惰清理（访问时检测过期）；  
    - 清理过程应优先采用增量/分批策略，避免集中清理导致明显的性能抖动或暂停。  
  - WAL 与快照管理：  
    - 持久化层应有明确的 WAL 截断与快照保留策略，例如“每次成功快照后截断对应区间 WAL，并保留最近 N 份快照 + WAL”；  
    - 在磁盘空间接近阈值时，应发出告警，并可以通过配置选择策略（如停止接受新写入、转为只读模式）。  
  - 合规与配置：  
    - 应提供与数据保留期相关的配置项（例如日志与快照的保留天数/份数），让部署方可根据合规要求进行调整；  
    - 文档中需明确说明 TokMesh 本身不会长期保留过期的 Session/Token 数据，避免对“历史会话检索”能力产生误解。  
- **验收视角 / 测试要点**：  
  - 在长时间运行和高并发写入场景下，内存与磁盘占用能保持在可预期范围内，不因过期数据堆积而线性增长；  
  - 在配置有限磁盘空间的环境中，通过模拟高写入与长时间运行，验证告警与保护策略（只读/拒绝写入）如预期工作。  
  - **实施阶段**：P1（基础清理能力）+ P2（完善保留策略与配置）

- **说明**：

### R28. 灾难恢复与跨集群备份策略

- **类型**：可靠性 / 运维 / 范围说明  
- **描述**：  
  - TokMesh v1 主要聚焦单集群内部的高可用与快速恢复，不在 v1 范围内提供完整的跨数据中心主动灾备方案，但需要对备份与恢复能力给出清晰边界与建议。  
- **关键约束 / 建议**：  
  - 单集群级别：  
    - 在 R11/R24 的基础上，通过快照 + WAL 保证单集群内节点重启与数据恢复能力，并可选支持将快照/备份定期同步到外部持久化存储（如对象存储）；  
    - 在文档中给出示例性的 RTO/RPO 目标区间（具体数值在 SDD-2/benchmark 中细化），并明确 TokMesh 只对“集群内部”的容错负责。  
  - 跨数据中心：  
    - v1 不提供内建的跨数据中心复制与自动故障切换机制；  
    - 如有跨 DC 容灾需求，应通过上层架构（例如双集群 + 外部流量调度 + 备份恢复）实现，并在部署文档中给出推荐拓扑与模式说明。  
- **验收视角 / 测试要点**：  
  - 在单集群范围内，通过模拟节点重启、磁盘故障等场景验证快照 + WAL 恢复路径的正确性与可接受的恢复时间；  
  - 明确文档中对跨数据中心容灾的声明与建议，避免对 v1 能力产生错误预期。  
  - **实施阶段**：P3（在集群化能力稳定后补齐灾备边界与建议）

### R17. 会话数据端到端加密存储（内存 + 落盘 + 高性能持久化）

- **类型**：安全 / 性能  
- **描述**：  
  - 与会话 / 令牌相关的敏感数据在 TokMesh 内部需要端到端加密存储：  
    - 在内存中遵循 R7 的字段级加密 / 混淆策略；  
    - 在落盘（快照、WAL、日志等持久化文件）时同样以加密形式保存敏感内容。  
  - 在保证安全性的前提下，持久化子系统需要维持高吞吐、低延迟的写入与恢复能力。  
- **关键约束 / 建议**：  
  - 加密实现仅依赖 Go 标准库或经过严格审计的密码库，避免自定义密码学原语。  
  - 所有持久化介质上的敏感字段必须加密存储，密钥不可明文落盘；密钥管理策略需在后续设计文档中单独说明（可对接操作系统密钥环或外部 KMS，但不强制）。  
  - 可以允许部分非敏感索引或统计数据保持明文，以兼顾查询性能，但必须有明确的“敏感字段划分规则”，保证不会通过这些字段轻易反推用户身份或令牌内容。  
  - 持久化路径需要通过专门的基准测试评估“加密前后”的性能变化，对写入 QPS、P99 延迟、恢复时间等核心指标做量化记录，并设定可接受阈值。  
  - 持久化方案优先采用顺序写、批量刷盘等方式，减少加密带来的 CPU / I/O 开销；必要时可以通过可配置的并行加密、异步刷盘等手段优化。  
  - 在设计与实现阶段，应补充一份专门的“加密与性能权衡”说明文档，记录不同策略下的安全收益与性能成本，便于后续版本迭代时进行调整。  
  - **实施阶段**：P2（加密增强：端到端加密存储）

### R20. Any-Node Gateway 与智能路由

- **类型**：架构 / 集群 / 体验  
- **描述**：  
  - TokMesh 集群应支持“任意节点网关（Any-Node Gateway）”能力：集群中任意存活节点都可以作为对外入口节点，接收业务或管理请求，并通过内部智能路由访问全局数据分片。  
  - 客户端（SSO/IAM/App/tokmesh-cli）无需了解分片与副本分布，只需连接一个或若干入口节点（或 LB），由 TokMesh 在集群内部完成请求转发与结果聚合。  
- **关键约束 / 建议**：  
  - 每个节点本地维护一份一致哈希 Ring 与“分区 → 副本节点列表”的路由表，并具备版本号（如 `ring_epoch`）以保证计算一致性。  
  - 对单 Key 操作（写入/校验/续期/撤销）：入口节点作为协调者，根据路由表将请求并发发送到对应副本节点，并按可配置的一致性级别（例如 `W=ONE|QUORUM|ALL` / `R=ONE|QUORUM`）汇总结果。  
  - 对全局或多 Key 查询/统计：入口节点可采用 scatter-gather 方式，将请求拆分到多个节点并在本地汇总结果，该类操作应优先通过管理端口（R19）暴露，并具备独立限流与超时控制。  
  - 客户端可以通过 LB 或 DNS 轮询访问多入口节点，但无需实现任何分片逻辑；TokMesh SDK 可以仅承担“多地址 failover”，而不负责智能路由本身（参见 R16）。  
  - **实施阶段**：P3（集群入口与路由能力；v1.0 可先以少量固定入口节点为主，逐步演进到完全对称）

### R21. 去中心化集群、副本数与节点重平衡

- **类型**：架构 / 集群 / 一致性  
- **描述**：  
  - TokMesh 集群应采用去中心化架构，各节点通过成员协议维护一致的集群视图，并使用一致哈希 Ring（建议配合虚拟节点）进行分片与副本分布管理。  
  - 支持可配置的全局或每租户副本数 `replication_factor = N`，每个分区在 Ring 上顺时针选择 N 个不同节点作为该分区的副本集合。  
  - 节点加入（join）或退出（leave/decommission）集群时，系统应自动触发数据重平衡，使数据和负载尽量均匀分布，并在过程中尽量保持读写可用与一致性约束。  
- **关键约束 / 建议**：  
  - 读写一致性：  
    - 写入/续期/撤销操作应支持以 `W=ONE|QUORUM|ALL` 形式配置写一致性级别；  
    - 读取/校验操作应支持以 `R=ONE|QUORUM` 形式配置读一致性级别；  
    - 推荐在默认配置下满足 `R + W > N` 以保障在无故障时的“读最新”语义，踢人/撤销等高风险操作的默认 `W` 值不应过低。  
  - 节点加入时的数据迁移：  
    - 新节点加入后，通过更新 Ring 引入新分片责任，并由原负责节点以“快照 + 增量日志”方式向新节点复制对应分区数据；  
    - 同步完成后，更新 Ring 版本并切换读写路径到新副本集合，避免长时间双写混乱。  
  - 节点下线时的数据迁移：  
    - 计划下线（graceful leave）应先将节点标记为 `LEAVING`，更新 Ring 后由其他节点接管其分区，并完成数据复制，再真正移除该节点；  
    - 非计划下线（故障）时，集群应在剩余副本足够的前提下尽量保持服务，对一致性不足的情况提供明确的策略（拒绝部分请求或允许配置化的一致性降级），并在故障节点恢复或替换后通过反熵机制补齐数据。  
  - 数据平均分配：  
    - 使用虚拟节点（vnode）提升分片分布的均匀性，减少单个节点加入/退出造成的数据倾斜；  
    - 后续可以在设计文档中扩展“按实际负载动态调整 vnode”和更精细的重平衡策略，但 v1 可先保证基本均衡与正确性。  
  - 集群时间一致性与健康：  
    - 集群节点应在启动与运行时持续监测本地时间与参考时间源（NTP 或等效方案）的偏差；  
    - 当单个节点的 Clock Skew 超过配置阈值时，应发出告警，并在必要时拒绝其加入或重新加入集群，以避免 TTL 与会话状态判断出现严重不一致；  
    - 集群健康检查与可观测性（结合 R22）中应包含对时间同步状态的暴露，便于运维发现和处理时钟漂移问题。  
  - **实施阶段**：P3（集群化核心能力；自动重平衡为中长期演进目标）

### R14. 打包与部署形态（Server/CLI/集群）

- **类型**：部署  
- **描述**：  
  - 构建产物统一：`tokmesh-server` 与 `tokmesh-cli`。  
  - 打包规范：

  - 支持的部署方式：

    - 容器化（Docker/Podman）与 K8S。  
- **关键约束 / 建议**：  
  - 运维操作尽量通过 CLI/API 提供统一入口（如节点 join/leave、状态查询）。  
  - 避免引入不必要的外部依赖（如额外数据库、中间缓存等）。
  - **实施阶段**：P1（MVP 部署形态）

### R15. 配置体验与使用手册

- **类型**：配置 / 文档  
- **描述**：  
  - 配置文件结构清晰、一致，字段有注释，对枚举类字段列出可选值及含义。  
  - 提供典型配置示例：  
    - 单节点（开发/测试）。  
    - 集群（生产）。  
    - Windows/Linux/容器环境的差异说明。  
  - 使用手册按“角色（开发/运维） + 操作系统 + 拓扑（单节点/集群）”组织。  
- **关键约束 / 建议**：  
  - 优先级略低于核心功能和安全，但必须纳入版本计划。  
  - 文档需与实际配置保持同步，避免“文档不可信”的情况。
  - **实施阶段**：P2（在 P1 能跑的基础上，逐步完善配置体验与文档）

## SDK 与接入体验类

### R16. 官方多语言 SDK 支持

- **类型**：接入 / 开发体验  
- **描述**：  
  - TokMesh 提供官方 SDK，简化常见 Web/后端语言对 TokMesh 的接入，优先支持：  
    - C#（ASP.NET，兼容 .NET Framework 与 .NET Core/.NET 5+）；  
    - Java；  
    - PHP；  
    - Python。  
  - SDK 封装对 TokMesh 的 HTTP/gRPC（或专用二进制协议）调用，提供统一的会话生命周期 API（结合 R4）以及可选的只读校验接口（结合 R10）。  
- **关键约束 / 建议**：  
  - SDK 应优先支持“服务端 SSO/IAM 使用场景”，默认集成对 mTLS 的支持，方便在生产环境中启用双向认证（关联 R5/R6）。  
  - SDK 需要提供清晰的接入文档和示例代码，覆盖上述语言的主流 Web 框架场景。  
  - 语言列表可在未来按使用反馈扩展或调整，但上述四种语言视为 v1 的重点。  
  - **关于缓存**：  
    - v1 SDK 不应在内部缓存 Token/Session 的“校验结果”，以避免与撤销/踢人语义冲突；  
    - 业务如需缓存，可在应用层自行以短 TTL 方式实现，并明确承担风险。  
  - **关于失败重试**：  
    - SDK 可以对“只读校验请求”（如 `ValidateToken`）在遇到明确的网络级错误（连接失败、超时）时，进行有限次数的自动重试（带最大次数与退避机制）；  
    - 对写入类请求（创建/续期/撤销）默认不做自动重试，或仅提供可选钩子，由 SSO/IAM 结合幂等设计自行控制。  
  - **关于负载均衡与路由**：  
    - v1 SDK 主要依赖运维侧配置的 LB 或 DNS 轮询来实现负载均衡；  
    - SDK 可选支持“多地址配置 + 简单 failover”（当前地址不可用时尝试下一个），而不实现复杂的权重/健康检查系统；  
    - 不在 SDK 内实现完整的负载均衡算法，避免与 LB/DNS 职责重叠并减少复杂度。
  - **实施阶段**：P4（多语言 SDK 与生态，在核心能力与集群化完成后实施）

### R18. 令牌类型与策略矩阵

- **类型**：功能 / 安全 / 策略  
- **描述**：  
  - TokMesh 在会话/令牌模型中应显式区分若干常见的令牌类型，并允许针对不同类型应用差异化的 TTL、索引与审计策略。  
  - 建议至少覆盖下列令牌类型，并在实现中以类似 `token_type` 字段加以标识：  
    - 短期访问令牌（Access / Session Token）：生命周期短、调用频率高、对延迟敏感，是主要校验对象；  
    - 长期刷新令牌（Refresh Token）：调用不频繁但一旦泄露危害大，需更强风控与审计；  
    - 设备绑定令牌（Device-Bound Token）：与设备指纹/客户端 ID 绑定，用于“记住设备”等场景；  
    - 一次性/短期操作令牌（One-Time / Action Token）：用于重置密码、敏感操作确认等，单次或极短 TTL；  
    - 高权限管理令牌 / 管理会话（Admin / Privileged Session）：对应 TokMesh 管理员、IAM 管理员等高权限主体；  
    - 服务间机器令牌（Service-to-Service / M2M Token）：用于服务间调用的机器身份。  
- **关键约束 / 建议**：  
  - 不同 `token_type` 应有明确的默认策略矩阵（在后续设计文档中表格化给出），包括：  
    - 默认 TTL 区间；  
    - 是否参与多维索引（User/IP/Device/Tenant 等）以及索引粒度；  
    - 审计级别（例如管理/高权限令牌需要更细的审计与告警能力）；  
    - 是否允许多点登录/多设备并存，以及踢人/撤销的语义；  
    - 对加密与存储的额外要求（例如刷新令牌、高权限令牌应视为最高敏感级别）。  
  - 令牌类型本身不直接暴露给终端用户，而是由 SSO/IAM/应用侧根据业务场景选择合适类型并通过 API/SDK 明确传递。  
  - 在迁移旧系统（使用 Redis 等）时，如原系统未显式区分令牌类型，应提供默认映射策略，以便平滑迁移但鼓励在后续版本中逐步引入更精细的类型划分。
  - **实施阶段**：P2（在 P1 Session/Token 模型基础上补全令牌类型策略）

### R19. 业务端口与管理端口隔离

- **类型**：安全 / 部署 / 性能  
- **描述**：  
  - TokMesh 应在监听端口层面区分业务数据平面与管理控制平面：  
    - 业务端口：承载会话生命周期 API（R4）与 App 只读校验 API（R10），主要面向 SSO/IAM 与业务系统；  
    - 管理端口：承载管理与统计相关 API（节点状态、配置变更、证书轮换、黑白名单管理、查询统计等），主要由 tokmesh-cli 与 IAM 管理后台使用。  
  - 可观测性相关接口（指标、健康检查等）可以根据部署策略复用业务端口或管理端口，但推荐靠近管理侧。  
- **关键约束 / 建议**：  
  - **安全隔离**：  
    - 管理端口必须部署在受控网段或通过 VPN/堡垒机访问，强制使用 mTLS 与强身份认证；  
    - 业务端口面向 SSO/IAM 与 App，可使用 TLS 或 mTLS（SSO/IAM 场景建议 mTLS，App 场景使用 TLS + 应用级鉴权），不应暴露管理操作。  
  - **性能与资源隔离**：  
    - 管理端口上的查询统计可能较“重”，应具备独立的限流、并发上限与超时配置，避免影响业务端口的低延迟需求；  
    - 可以为业务端口和管理端口分别暴露不同的监听地址（IP + 端口）与线程/协程池配置，便于按需调优。  
  - **运维与监控**：  
    - 日志与指标中应能按端口维度区分业务流量与管理流量，便于分析与告警；  
    - 允许在运维场景下临时关闭管理端口（维护/排障）而尽量不影响业务端口对外提供校验服务。
  - **实施阶段**：P1（MVP 端口与平面隔离）

### R26. 配额与限流机制（IP / Client / API）

- **类型**：可靠性 / 安全 / 资源保护  
- **描述**：  
  - 为防止单一来源或租户滥用接口资源，TokMesh 需要在节点层面提供基础配额与限流能力，可与上游 LB/WAF 策略协同工作。  
- **关键约束 / 建议**：  
  - 限流维度：  
    - 至少支持按 IP 的基础限流（与 R8 中的防御策略呼应）；  
    - 支持按 Client_ID（或等效身份标识）配置限流与配额，用于区分不同接入方/应用的资源使用；  
    - 按 API 端点区分限流策略，例如对只读校验、生命周期写入、管理操作使用不同的默认与上限。  
  - 配额与超限行为：  
    - 限流与配额应通过配置驱动，允许为不同环境（开发/测试/生产）设置不同阈值；  
    - 当达到或超过配额/限流阈值时，应返回明确的错误（例如 HTTP 429 或等效错误码），并避免对后端资源造成进一步压力。  
  - 与外部组件协同：  
    - 文档中应明确推荐：TokMesh 内建限流用于“最后一道保护”，上游 LB/WAF 可承担更细粒度和更高 QPS 的限流职责；  
    - 通过 R22 中的指标暴露限流/配额触发情况，便于运维在外部系统中调整策略。  
- **验收视角 / 测试要点**：  
  - 在高并发单 IP 或单 Client_ID 压力测试下，验证限流与配额机制能够有效限制总 QPS，不让单一来源拖垮节点；  
  - 在不同端点（只读校验 / 写入 / 管理）上验证限流策略可独立配置并生效。  
  - **实施阶段**：P2（在基础功能与安全能力落地后逐步完善）

### R27. 运行时日志与诊断（Application Logs）

- **类型**：可靠性 / 可观测性 / 运维  
- **描述**：  
  - TokMesh 需要除审计日志（R23）外的运行时日志，用于开发调试、故障排查与性能分析，包括启动关闭、错误与异常、慢请求、资源水位与后台任务等场景。  
- **关键约束 / 建议**：  
  - 日志级别与分类：  
    - 至少支持常见日志级别（DEBUG / INFO / WARN / ERROR / FATAL），并可按模块或全局配置；  
    - 区分审计日志与运行时日志：审计侧聚焦“谁在何时做了什么”，运行时日志聚焦“系统发生了什么、状态如何”。  
  - 输出与轮转：  
    - 支持将运行时日志输出到 stdout（容器化友好）和/或本地文件，并提供按大小或时间的轮转策略；  
    - 日志格式应为结构化（例如 JSON）或至少包含固定字段顺序，便于集中收集与查询。  
  - 关联性与采样：  
    - 为请求流量引入 `request_id`/`trace_id`（可复用 OpenTelemetry 风格），所有与同一请求相关的日志应携带相同标识；  
    - 对高频重复日志（如相同错误）可支持采样或去重策略，避免日志风暴对性能和磁盘造成过大压力。  
  - 敏感字段与加密：  
    - 与 R23 一致，运行时日志中涉及的高敏感字段应避免直接明文记录或通过脱敏处理；  
    - 对落盘运行时日志是否加密可视部署要求配置，默认可不加密，但需在文档中给出安全建议。  
- **验收视角 / 测试要点**：  
  - 在启动、关闭、典型错误、慢请求与资源异常场景下，运行时日志能提供足够上下文用于排查；  
  - 在高 QPS 场景下启用 INFO/WARN/ERROR 级日志时，不会对核心路径性能造成不可接受的影响（具体指标在 SDD-2/实现阶段通过 benchmark 给出）。  
  - **实施阶段**：P2（在 P1 基础能力可运行的前提下，补齐系统级诊断能力）

### R29. 版本兼容性与升级策略

- **类型**：运维 / 范围说明 / 约束  
- **描述**：  
  - TokMesh v1 需要对持久化格式与对外 API 的演进给出基本的兼容性与升级约束，避免在滚动升级或版本迁移中出现不可预期的行为。  
- **关键约束 / 建议**：  
  - 持久化格式（WAL / 快照）：  
    - 为持久化数据结构引入显式的版本号（例如 `format_version`），新版本在可行范围内应尽量向后兼容旧版本数据，或提供明确的迁移/重建路径；  
    - 禁止在小版本升级中悄然引入不兼容的持久化格式变更，若需破坏性变更，应在文档与发布说明中明确标记，并提供升级指引。  
  - 对外 API（HTTP/gRPC）：  
    - 建议采用显式的 API 版本前缀（如 `/api/v1/...`），在引入破坏性变更时通过新版本路径发布，旧版本在一段兼容期内保留；  
    - Redis 子集协议的命令集扩展应视为向后兼容变更，不得在已有命令语义上做破坏性修改；  
    - gRPC API 推荐通过 Protobuf 的 `reserved` 字段与 `deprecated` 标记来演进消息结构，服务端在兼容期内应保留对旧字段的处理能力；  
    - 错误码与响应结构的变更应通过版本化或兼容字段扩展实现，避免对现有客户端造成不可预期的解析失败。  
  - 内部 API（SSO/IAM 调用）：  
    - 与对外 API 共享相同的版本化策略，避免出现“内部调用版本与公开 API 版本脱节”的情况；  
    - SDK（R16）应封装版本协商逻辑，使客户端可以在不修改业务代码的前提下平滑适配服务端的兼容版本升级。  
  - 滚动升级与多版本共存：  
    - 集群在滚动升级期间应支持短时间内不同版本节点共存，但需要在 SDD-2 中定义支持的版本跨度（例如 N 个小版本以内），并在实现上避免出现协议/格式冲突；  
    - 建议在运维手册中提供“升级步骤示例”（先升级控制面/Seed Nodes，再升级普通节点等），以降低操作风险。  
- **验收视角 / 测试要点**：  
  - 在单节点与小规模集群上，验证从旧版本升级到新版本时 WAL/快照与网络协议不会导致数据丢失或严重服务中断；  
  - 检查 API 版本化策略与发布说明是否清晰描述了兼容性边界与破坏性改动。  
  - **实施阶段**：P4（在核心能力稳定后补齐长期演进约束）

### R22. 可观测性与监控指标（Prometheus）

- **类型**：可靠性 / 可观测性 / 运维  
- **描述**：  
  - TokMesh 应提供系统级可观测性能力，包括 Prometheus 指标、基础健康检查与必要的节点状态查询接口。  
  - 指标与健康检查默认通过管理控制平面暴露，便于集中运维和权限控制。  
- **关键约束 / 建议**：  
  - 指标暴露：  
    - 提供符合 Prometheus 规范的 `/metrics` HTTP 端点，覆盖核心维度：节点资源（CPU、内存、GC）、会话/令牌数量与变化、请求 QPS 与延迟直方图、错误率、安全事件计数（踢人、防御触发等）。  
    - `/metrics` 应默认通过管理端口暴露，并支持 TLS/mTLS（可通过配置 CA 与客户端证书，让 Prometheus 以 mTLS 抓取）。  
  - 标签与基数控制：  
    - 所有指标统一使用 `tokmesh_` 前缀；  
    - 标签限定在 `node_id`、`cluster_id`、`endpoint`、`result`、`plane`（业务/管理）等有限维度，禁止使用 `user_id`、`device_id` 等高基数字段作为标签，以控制内存与 Cardinality。  
  - 健康检查：  
    - 提供 liveness / readiness / health 类接口，便于 K8s / LB 使用；  
    - readiness 应综合至少：存储子系统可用性、核心后台任务状态，以及在集群模式下的成员视图是否稳定。  
  - 版本与兼容性：  
    - 指标名和标签视为外部 API；变更需要在 Release Note 中记录，并预留兼容期（例如同时暴露旧/新指标一段时间）。  
- **验收视角 / 测试要点**：  
  - 在单节点和集群部署中，通过 Prometheus 成功抓取 `/metrics`，并验证关键指标（QPS、延迟、资源水位）随负载变化合理变动。  
  - 在启用 mTLS 的管理端口上，验证 Prometheus 配置客户端证书后仍能稳定抓取指标。  
  - 在节点异常（高内存、持久化失败、部分节点故障）场景下，相关错误/告警类指标应有可观测变化，便于构建告警规则。  
  - **实施阶段**：P2（在 P1 能力基础上补齐可观测性与监控）
