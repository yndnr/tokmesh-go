## 二、新增需求（R28/R29）的问题

### 问题 4：R28（灾难恢复）与 R11/R24 边界模糊 ⚠️

**位置**：行 368-383

**问题描述**：
- **R28** 定义为"灾难恢复与跨集群备份策略"，但其内容与 **R11（会话持久化与快速恢复）** 和 **R24（数据保留与清理策略）** 高度重叠
- R28 强调的"单集群快照备份到外部存储"实际上是 R11/R24 的自然扩展，而非独立需求

**具体重叠**：
| 需求 | 核心内容 | 重叠点 |
|------|---------|--------|
| R11 | 会话持久化（WAL + 快照）、节点重启快速恢复 | 单节点/集群节点的快照能力 |
| R24 | WAL 截断、快照保留策略、磁盘空间管理 | 快照管理策略 |
| R28 | 单集群快照备份、RTO/RPO 目标、跨 DC 声明 | "快照同步到外部存储"是 R24 的扩展 |

**影响程度**：🟡 **中**（需求边界不清晰，可能导致实现时职责混乱）

**修复建议**：
```markdown
# 方案 1（推荐）：将 R28 合并到 R24
### R24. 数据保留、清理与备份策略（内存 + 持久化 + 外部备份）
- 在 R24 中补充"外部备份"相关约束：
  - 支持将快照定期同步到对象存储（如 S3/MinIO）
  - 提供明确的 RTO/RPO 目标区间（在 SDD-2 中细化）
  - 跨数据中心容灾不在 v1 范围内，应通过上层架构实现

# 方案 2：保留 R28，但明确与 R11/R24 的关系
- 在 R28 中增加"依赖 R11 的 WAL + 快照能力，在 R24 的保留策略基础上..."
- 避免重复定义已有能力
```

### 问题 5：R29（版本兼容性）缺少与 R13/R4 的对齐 ⚠️

**位置**：行 622-640

**问题描述**：
- **R29** 定义了持久化格式与 API 的版本兼容性约束，但：
  1. **未明确多协议网关（R13）的版本兼容策略**：
     - Redis 协议子集的命令集是否会扩展？
     - HTTP/gRPC API 的版本化策略具体如何（如 `/api/v1` vs `/api/v2`）？
  2. **未明确内部 API（R4）的演进约束**：
     - SSO/IAM 使用的内部 API（如 `CreateSession`）是否独立版本化？
     - 还是与 HTTP/gRPC 公共 API 共享版本号？

**影响程度**：🟡 **中**（升级策略不完整，可能影响 SSO 系统的兼容性）

**修复建议**：
```markdown
# 在 R29 中补充以下约束：

- **多协议网关版本化（R13）**：
  - Redis 协议子集：命令集扩展视为"向后兼容的增量变更"，旧客户端不受影响；
  - HTTP API：采用 `/api/v1/...` 路径前缀，破坏性变更通过新版本路径（如 `/api/v2/...`）发布；
  - gRPC API：通过 Protobuf 的 `reserved` 字段与 `deprecated` 标记实现演进，服务端保留旧版本消息处理能力。

- **内部 API 演进（R4）**：
  - 内部 API（SSO/IAM 调用）与外部 API（App 只读校验）使用相同的版本化策略；
  - 在 SDK（R16）中封装版本协商逻辑，SDK 默认使用最高版本 API，但可兼容低版本服务端。
```

### 问题 8：R3 时间容忍窗口与 R21 Clock Skew 阈值未对齐 ⚠️

**位置**：R3（行 114-115）、R21（行 437-438）

**问题描述**：
- **R3** 中提到 TTL 判断的"容忍窗口"：
  > 在 TTL 判断等边界场景可预留小幅容忍窗口（例如 ±30 秒，具体数值在 SDD-2 中细化）

- **R21** 中提到集群时间同步的"最大 Clock Skew"：
  > 当单个节点的 Clock Skew 超过配置阈值时，应发出告警，并在必要时拒绝其加入或重新加入集群

- **问题**：
  1. R3 的"容忍窗口 ±30 秒"与 R21 的"最大 Clock Skew ≤ 60 秒"（来自之前的 R3 约束）是否一致？
  2. 如果 Clock Skew = 60 秒，TTL 容忍窗口 = ±30 秒，是否意味着最差情况下两个节点对同一会话的 TTL 判断可能相差 90 秒？

**影响程度**：🟡 **中**（可能导致集群模式下会话有效性判断的不一致）

**修复建议**：
```markdown
# 在 R3 中明确容忍窗口与 Clock Skew 的关系：
- 时间语义需可预期：
  - 所有与 TTL、创建时间、最后活动时间相关的判断统一基于 UTC 时间；
  - 在集群模式下，生产环境必须配置 NTP 或等效时间同步机制，并约定最大允许 Clock Skew（建议 ≤ 60 秒，可配置）；
  - TTL 判断的容忍窗口应 **小于等于最大 Clock Skew 的一半**（例如 Clock Skew ≤ 60 秒时，容忍窗口 ≤ ±30 秒），以确保在最差情况下不同节点对同一会话有效性的判断偏差可控；
  - 具体数值在 SDD-2 中细化，并通过配置项统一控制。
```

## 四、遗漏点与建议

### 建议 1：补充"会话/令牌的最大数量限制"需求 💡

**当前状态**：
- **R12** 定义了"内存阈值与节点资源管理"，关注内存占用
- **R2** 定义了"多维索引"，但未明确索引数据结构的内存开销上限

**问题**：
- 如果攻击者创建大量短 TTL 会话（例如每秒 10K 个，TTL=1 秒），即使单个会话很小，累积的索引开销可能拖垮节点
- R12 的"内存阈值"是被动保护，无法在攻击初期就拦截异常行为

**建议**：
```markdown
# 新增需求（可作为 R25）：
### R25. 会话与令牌数量限制（租户 / 全局）

- **类型**：可靠性 / 安全
- **描述**：
  - TokMesh 应支持配置全局或按租户的会话/令牌数量上限，防止单一租户或全局会话数膨胀导致资源耗尽。
- **关键约束 / 建议**：
  - 全局限制：节点级别的最大会话总数（如 1,000,000），超过后拒绝新会话创建；
  - 租户限制：每个 Tenant 的最大会话数（如 100,000），超过后该租户无法创建新会话；
  - 配额与 R26 的限流机制协同：限流控制"速率"，配额控制"总量"；
  - 在接近限制时（如达到 80%），通过 R22 的指标与 R27 的日志发出告警。
- **实施阶段**：P2（在资源管理能力基础上补齐数量限制）
```

### 建议 2：明确"会话续期"的幂等性约束 💡

**当前状态**：
- **R3** 提到"原子续期（Replay-Extension）"，但未明确续期操作的幂等性语义
- **R4** 定义了 `ExtendSession` API，但未说明重复续期请求的行为

**问题**：
- 如果客户端因网络超时重试 `ExtendSession`，是否会导致 TTL 被多次延长？
- 集群模式下，如果续期请求被路由到不同节点，如何保证一致性？

**建议**：
```markdown
# 在 R3 中补充：
- 生命周期语义要严谨，避免并发续期、过期边界问题：
  - **续期幂等性**：同一 SessionID 的续期请求应基于"最后活动时间"或"请求时间戳"进行幂等判断，避免短时间内重复续期导致 TTL 累加；
  - 具体策略在 SDD-2 中细化（例如"续期操作将 TTL 重置为固定值"或"基于请求时间戳判断是否在容忍窗口内"）。
```

---

### 建议 3：补充"集群分裂（Split-Brain）"处理策略 💡

**当前状态**：
- **R21** 定义了"去中心化集群与副本数"，提到了节点加入/退出的数据迁移
- 但未明确在网络分区（Network Partition）导致集群分裂时的行为

**问题**：
- 如果集群因网络分区分裂为两个子集群（如 3 节点集群分裂为 2+1），是否允许两边同时接受写入？
- 如何避免数据不一致与会话状态冲突？

**建议**：
```markdown
# 在 R21 中补充：
- **集群分裂（Split-Brain）处理**：
  - 在发生网络分区时，TokMesh 应采用"多数派可用"策略：仅当子集群节点数 ≥ (总节点数 / 2 + 1) 时才接受写入；
  - 少数派子集群应进入"只读模式"或"拒绝写入模式"，避免数据分叉；
  - 网络恢复后，通过反熵机制与版本向量（Vector Clock）或类似机制检测并解决冲突；
  - 具体冲突解决策略在 SDD-2 中细化（例如"最后写入胜出（LWW）"或"拒绝冲突写入"）。
- **实施阶段**：P3（集群化核心能力）
```
