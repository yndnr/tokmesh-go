# TokMesh 需求收件箱（Backlog Inbox）

> **用途**：临时承载尚未正式拆分为 RQ-ID 的“零散想法 / 草稿需求”。  
> **规则**：这里只存放「待归档」的想法；一旦被整理进正式 RQ（并写入 SDD-1），就从这里标记为已处理或移除。

## 使用方式

- 当有新的想法、补充点、零散需求时：  
  - 先在本文件中新增一条 Inbox 条目（由 AI 或人工填写）。  
  - 定期由“虚拟架构师 + 合规审查员”角色梳理，将条目归入具体 RQ 或拆成新 RQ。  
- registry (`docs/sdd/registry.md`) 只列「正式 RQ」，不直接记录这些零散条目。

## Inbox 条目模板

> 新增条目时，复制以下模板，并将状态初始设为 `pending`。

```markdown
### INBOX-YYYYMMDD-XX：简短标题

- **状态**：pending / merged / rejected  
- **来源**：谁提出、来自哪次讨论（可选）  
- **简要描述**：  
  - 用 2～3 句话说明想法或需求的核心内容。  
- **初步归属判断**（可由 AI 填写）：  
  - 可能关联的 RQ-ID：例如 RQ-TM-000X-...（如未确定，可写“待定”）  
  - 属于哪一类：功能 / 安全 / 持久化 / 部署 / 配置 / 文档 / 其他  
- **开放问题 / 待澄清点**：  
  - 条目 1  
  - 条目 2  
```

### INBOX-CRYPTO-PERF-001：全链路加密与高性能的权衡（R7/R17）

- **状态**：pending  
- **来源**：外部 AI 审查意见（Critical Risks #1）  
- **简要描述**：  
  - 现有设计要求对内存和落盘数据进行端到端加密（R7、R17），但在高并发 SSO 场景（例如 100k+ TPS）下，每次读写都要进行加解密，CPU 可能成为瓶颈。  
  - 文档中尚未明确加密算法选择（如 AES-GCM vs Chacha20-Poly1305），也未细化哪些字段必须加密，哪些可以明文以降低成本。  
- **初步归属判断**：  
  - 可能关联的 R：R7、R17、R11（持久化性能）、整体性能红线要求。  
  - 类别：安全 / 性能 / 持久化。  
- **开放问题 / 待澄清点**：  
  - 是否需要为不同字段定义加密等级（如 User_ID/Token_Payload 强制加密，TTL/Last_Active_Time 可明文）？  
  - 是否需要在 SDD-1 中明确推荐加密算法与是否依赖硬件加速（AES-NI）？  
  - 在无硬件加速场景下是否允许通过配置降低加密强度或范围？

### INBOX-KICK-CLUSTER-002：分布式环境下按 User O(1) 踢人的语义（R2）

- **状态**：pending  
- **来源**：外部 AI 审查意见（Critical Risks #2）  
- **简要描述**：  
  - R2 承诺“O(1) 或近似 O(1)”的踢人能力，但在 P3 集群模式下，会话数据被分片到多个节点，按 UserID 踢掉所有设备上的会话在现实上更接近 Scatter-Gather 操作。  
  - 若不引入全局二级索引，按非分区键（UserID）踢人很难做到真正 O(1)。  
- **初步归属判断**：  
  - 关联 R：R2、R20、R21。  
  - 类别：功能 / 集群 / 一致性。  
- **开放问题 / 待澄清点**：  
  - 是否需要在 R2 中明确：“按 SessionID 踢人”为 O(1)；“按 UserID 批量踢人”为 O(N_nodes) 或依赖某种全局索引？  
  - Partition Key 是否需要强制包含 UserID？如包含，如何缓解单用户热点集中问题？  
  - 是否需要在集群设计中限制某些批量踢人操作的频率或范围？

### INBOX-TIME-CLOCK-003：集群时钟同步与 TTL/时间语义（Cluster Time）

- **状态**：pending  
- **来源**：外部 AI 审查意见（Critical Risks #3）  
- **简要描述**：  
  - Session/Token 系统对时间非常敏感（TTL、签发时间、Last Active Time）。当前集群设计文档对节点间时钟同步（NTP 或逻辑时钟、Clock Skew 容忍窗口）未作说明。  
  - 节点时钟偏差会导致不同节点对同一 Token 的有效性判断不一致。  
- **初步归属判断**：  
  - 关联 R：R3（生命周期）、R11（恢复语义）、R20/R21（集群设计）。  
  - 类别：可靠性 / 集群 / 时间语义。  
- **开放问题 / 待澄清点**：  
  - 是否要求部署环境必须配置 NTP，并定义最大允许 Clock Skew？  
  - 是否需要在协议层为时间相关判断预留容忍窗口（例如 ±几秒/分钟）？  
  - 是否需要在审计/日志中记录节点本地时间与参考时间的偏差？

### INBOX-APP-AUTH-004：App 直连 TokMesh 校验的鉴权机制（R10）

- **状态**：pending  
- **来源**：外部 AI 审查意见（Gaps #1）  
- **简要描述**：  
  - R10 允许 App 直连 TokMesh 做只读校验，文档强调 TLS，但尚未明确 App 端自身的鉴权机制。  
  - 如果仅凭 Token 即可发起校验，恶意客户端可能对 Token 空间进行暴力枚举或撞库。  
- **初步归属判断**：  
  - 关联 R：R10、R8、R19。  
  - 类别：安全 / 接入。  
- **开放问题 / 待澄清点**：  
  - 是否要求 App 侧使用 Client_ID / API_Key / 签名机制进行鉴权？  
  - 是否需要配合 IP 段白名单、LB/WAF 限流等措施，并在文档中明确“TokMesh 内部做基础限流，外侧由 LB/WAF 承担大部分防护”？  
  - 该鉴权模型是否与现有 SSO/IAM 的 Client 模型对齐？

### INBOX-REDIS-MTLS-005：Redis 子集协议与 mTLS 的迁移兼容性（R13 vs R6）

- **状态**：pending  
- **来源**：外部 AI 审查意见（Gaps #2）  
- **简要描述**：  
  - R13 希望通过 Redis RESP 子集支持旧系统迁移，R6 又强调生产环境 mTLS。现实中大量 Redis 客户端对 mTLS 支持薄弱，通常需借助 Stunnel/Sidecar 等。  
  - 强制 Redis 通道 mTLS 可能严重增加迁移摩擦，违背“平滑迁移”的初衷。  
- **初步归属判断**：  
  - 关联 R：R6、R13。  
  - 类别：协议 / 安全 / 迁移体验。  
- **开放问题 / 待澄清点**：  
  - 是否允许为 Redis 子集协议定义“受控网段 + TLS 或密码认证”的过渡模式，而不是立即强制 mTLS？  
  - 是否倾向通过 Sidecar/Proxy 为 Redis 流量补上 mTLS，而 TokMesh 自身始终以 TLS/mTLS 为内部标准？  
  - 迁移阶段的“安全下限”如何在文档中清晰定义？

### INBOX-KEY-ROTATION-006：密钥轮换与落盘数据的处理策略（R9 vs R17）

- **状态**：pending  
- **来源**：外部 AI 审查意见（Gaps #3）  
- **简要描述**：  
  - R9 提到证书/通讯密钥轮换，R17 提到持久化数据加密，但尚未明确主密钥（Master Key）轮换时对存量加密数据的策略：是否重加密所有历史快照/WAL，还是采用 Key Versioning 策略。  
- **初步归属判断**：  
  - 关联 R：R7、R9、R11、R17。  
  - 类别：安全 / 持久化 / 运维。  
- **开放问题 / 待澄清点**：  
  - 是否采用 Key Versioning（为每份快照/WAL 记录密钥版本，旧数据由旧密钥解密直到自然过期），避免大规模重加密引发 IO 风暴？  
  - Session TTL 通常较短，是否可以在文档中明确“不对历史数据强制重加密，依赖 TTL 淘汰”的策略？  
  - 主密钥失效/丢失时的失败模式与恢复策略如何描述？

### INBOX-CLUSTER-ROUTE-007：入口节点转发时的内部通道选择（Cluster Routing）

- **状态**：pending  
- **来源**：外部 AI 审查意见（Cluster Risk）  
- **简要描述**：  
  - R20/R21 与集群设计文档提出 Any-Node Gateway，入口节点接收外部业务流量后，会将请求转发给内部其它节点。  
  - 目前尚未在文档中明确：入口节点对内转发使用的是独立的、受控的 mTLS 内部通道，还是复用对外业务端口，这会影响安全域划分。  
- **初步归属判断**：  
  - 关联 R：R6、R19、R20、R21。  
  - 类别：安全 / 集群 / 网络架构。  
- **开放问题 / 待澄清点**：  
  - 是否强制要求“外部流量只进业务端口，集群内部转发走独立内部端口或内部网络接口（mTLS）”？  
  - 是否需要在配置中区分“外部监听地址/端口”和“内部集群通讯地址/端口”？  
  - 在多网卡/多子网环境下如何文档化推荐部署拓扑？
