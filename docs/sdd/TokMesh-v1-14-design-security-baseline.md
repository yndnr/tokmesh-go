## 1. 安全目标与威胁模型（概览）

- **安全目标**：  
  - 保护会话/令牌数据在传输、内存和落盘时不被轻易窃取或篡改。  
  - 对高权限操作和管理接口进行强身份认证与严格审计。  
  - 在遭遇部分节点入侵或磁盘泄露时，限制攻击者可获得的信息和可操作范围。  

- **威胁模型简述**：  
  - 假设攻击者可能：  
    - 嗅探网络流量；  
    - 获取某节点磁盘（快照、WAL）；  
    - 在一定范围内获得进程内存 dump；  
  - 不承诺对拥有长期 root 权限且能持续控制环境的对手提供“绝对安全”，而是专注于提升攻击成本与缩短可利用窗口。

## 2. 证书与 PKI 基线（R5/R6/R9）

- **证书角色**：  
  - 集群 CA（TokMesh 内部或企业 PKI）：签发节点证书与客户端证书。  
  - 节点证书（Server Cert）：供 tokmesh-server 在业务/管理/内部端口上使用。  
  - 客户端证书（Client Cert）：供 tokmesh-cli、SSO/IAM 服务使用（mTLS 场景）。  

- **证书存放与权限**：  
  - 默认使用 PEM 编码证书与私钥文件，存放于可配置目录（如 `/etc/tokmesh/certs`）。  
  - 私钥文件仅 tokmesh-server 进程用户可读；tokmesh-cli 在本地环境使用自己的证书与私钥。  

- **mTLS 使用范围**：  
  - 管理控制平面、集群内部通讯：生产环境中必须启用 mTLS。  
  - SSO/IAM 调用生命周期 API：推荐启用 mTLS，至少需要 TLS + 强鉴权。  
  - App 只读校验 API：TLS 必须，mTLS 可按场景决定（结合 LB/WAF 等）。  

- **证书轮换基线（与 R9、INBOX-KEY-ROTATION-006 关联）**：  
  - tokmesh-cli 提供基础证书生成/导入与更新能力；  
  - 节点支持多版本证书并可回滚；  
  - 对于数据加密密钥，采用 Key Versioning 策略，避免全量重加密。

## 3. 端口与安全域划分（R19，补足集群内部路由问题）

- **业务端口**：  
  - 用途：生命周期 API（R4）、App 只读校验 API（R10）、业务侧 HTTP/gRPC/Redis 接入。  
  - 安全要求：  
    - 外部暴露时必须使用 TLS；  
    - 仅 SSO/IAM 与业务 App 通过业务端口访问，不暴露管理操作；  
    - 可结合 LB/WAF 做额外访问控制与限流。  

- **管理端口**：  
  - 用途：tokmesh-cli 与 IAM 管理后台调用管理 API（配置、证书、人为踢人、统计等）。  
  - 安全要求：  
    - 必须使用 mTLS；  
    - 部署在受控网段或通过 VPN/堡垒机访问；  
    - 所有高权限操作需有审计记录。  

- **集群内部通讯端口**：  
  - 用途：节点间路由、心跳、分片迁移与一致性协议。  
  - 安全要求（补足 INBOX-CLUSTER-ROUTE-007）：  
    - 必须独立于外部业务/管理端口配置，走内部 mTLS 通道；  
    - 入口节点在接收外部请求后，仅通过内部通讯端口向其他节点转发，不得通过对外业务端口“反向转发”，避免安全域混淆。

## 4. App 直连校验安全（R10，补足 INBOX-APP-AUTH-004）

- **传输层**：  
  - 对外暴露的只读校验 API 必须通过 HTTPS（TLS）提供。  

- **调用方鉴权**：  
  - App 调用校验 API 时，应携带：  
    - Token/Session 信息；  
    - App 身份信息：如 Client_ID + API_Key 或 HMAC 签名。  
  - TokMesh 对 App 身份做基础鉴权与限流，避免恶意客户端对 Token 空间进行大规模枚举。  

- **与外部防护配合**：  
  - 对大流量防刷/防 DDoS：建议主要由 LB/WAF 实现；  
  - TokMesh 端重点关注按 Client_ID、IP 做细粒度限流与黑/白名单控制。  

---

## 5. 主动防御与黑白名单（R8）

- **防重放**：  
  - 对校验与生命周期操作设置合理的重放检测机制（如短期窗口内的重复请求检测）。  

- **黑名单**：  
  - 支持按 IP、Client_ID、UserID 等维度加入黑名单；  
  - 提供解封与回滚机制，避免误伤难以恢复。  

- **白名单与豁免 IP**：  
  - 支持对特定监控/管理来源放宽部分限流或防御策略。  

- **配置与可观测性**：  
  - 防御策略的阈值可配置；  
  - 防御触发与封禁事件必须可审计与可观测（计数器/日志）。  

---

## 6. 时间与时钟同步（与 INBOX-TIME-CLOCK-003 关联）

- **时间敏感性**：  
  - TTL、签发时间、last_active_at 以及审计都严重依赖可靠的时间源。  

- **环境要求**：  
  - 部署环境应启用 NTP 或等效时间同步机制；  
  - 在文档中建议最大允许 Clock Skew（例如几秒到几十秒范围内）。  

- **应用层容忍**：  
  - 在生命周期逻辑与判断逻辑中预留小的容忍窗口（Grace Window），避免微小时间偏差导致跨节点判断不一致。
