# TokMesh v1 集群与一致性设计草案（SDD-0 草案）

> **基本信息**：本文件在 R11、R20、R21 等需求基础上，给出 TokMesh v1 集群形态、智能路由与副本一致性的整体思路，用于后续 SDD-1/SDD-2 细化与实现对齐。  
> **范围**：仅讨论单集群内部节点的分片、副本、一致性与路由机制，不涉及跨数据中心复制等高级主题。  
> **关联 R**：R11、R20、R21（必要时参考 R2/R3/R19/R22）

---

## 1. 背景与目标

- **背景**：TokMesh 需要在多节点环境下提供高可用、可横向扩展的会话存储能力，同时满足：  
  - 任意节点可作为入口（Any-Node Gateway）；  
  - 可配置副本数与读写一致性策略；  
  - 节点加入或下线时的数据重平衡与一致性保障。  
- **目标**：  
  - **G1：Any-Node Gateway**——任意存活节点都可以对外提供完整的业务/管理能力，通过内部智能路由访问全局分片数据。  
  - **G2：去中心化与副本**——集群不依赖单一中心节点，使用一致哈希 + 副本机制分布数据并提高可用性。  
  - **G3：可配置一致性**——对写入/续期/撤销与校验操作分别支持配置化的读写一致性级别，在性能与“踢人可靠性”之间取得平衡。  
  - **G4：平滑的 Join/Leave**——节点加入/下线时通过数据重平衡维持大致均衡的存储与负载，尽量降低对线上服务的扰动。

---

## 2. 集群成员与一致哈希 Ring

- **成员视图**：  
  - 通过成员协议或集中配置源维护“当前存活节点列表 + 节点权重”等信息。  
  - 每个节点本地维护一份带版本号的集群视图与一致哈希 Ring 元数据（例如 `ring_epoch`）。  
  - **Ring 变更治理（防止脑裂的基本约束）**：  
    - v1 建议采用“Seed Nodes + 版本号”模式：  
      - 集群在配置中指定一组 Seed Nodes（数量较少且稳定），由它们负责生成新的 Ring 版本与 `ring_epoch`；  
      - 其他节点通过成员协议与管理平面从 Seed Nodes 获取最新 Ring，而不是各自独立生成。  
    - Ring 变更流程（概念级）：  
      - 成员变化（节点加入/离开/权重变化）在成员协议中被观测到后，由某个 Seed Node 或 Seed 组内选出的协调者提议新的 Ring；  
      - 新 Ring 配置带有单调递增的 `ring_epoch`，通过管理平面或 Gossip 传播至全体节点；  
      - 节点在进行路由计算时应拒绝使用未知或回退的 `ring_epoch`，避免本地单独“造 Ring”。  
    - 完全 Gossip 自生成 Ring 的模式（类似部分无中心系统）暂不作为 v1 的默认路径，后续可在开放问题中评估。

- **一致哈希 Ring 与虚拟节点**：  
  - 将哈希空间（如 [0, 2^M)）划分给若干虚拟节点（vnode），每个物理节点关联多个 vnode，以减少单节点加入/退出带来的数据倾斜。  
  - 分区键（推荐 `session_id` 或内部 `partition_id` 哈希）映射到哈希空间，再根据 Ring 找到负责该分区的 vnode 与物理节点。  
  - **副本数 `replication_factor = N`**：对每个分区，在 Ring 上顺时针选取 N 个不同节点形成副本集合。

## 3. Any-Node Gateway 与智能路由

- **入口节点（Gateway 节点）**：  
  - 集群中任意存活节点都可以作为入口，分别在业务端口和管理端口上对外提供服务（关联 R19）。  
  - 客户端（SSO/IAM/App/tokmesh-cli）只需知道一个或若干入口地址，TokMesh SDK 主要负责多地址 failover，而不承担分片计算职责（关联 R16）。  

- **单 Key 操作的智能路由**：  
  - 入口节点接收到操作（创建/校验/续期/撤销）时：  
    - 使用当前 `ring_epoch` 的一致哈希 Ring 将 key 映射到分区；  
    - 根据副本数获取该分区的副本节点列表 `[P0, P1, ... P(N-1)]`；  
    - 按配置的写一致性级别 `W` 将请求并发转发给多个副本节点，并等待对应数量的响应；  
    - 对读操作则按读一致性级别 `R` 决定读取多少副本并合并结果。  

- **多 Key / 全局查询的 scatter-gather**：  
  - 对于管理统计类操作（例如全局会话数、按租户聚合等）：  
    - 入口节点将请求拆分成多个子请求，分发到不同节点（按分区或节点维度划分）；  
    - 各节点本地计算部分结果后返回，入口节点汇总并返回给客户端。  
  - 该类操作只应通过管理端口暴露，并配合独立的限流和超时策略（参见 R19）。

## 4. 副本与读写一致性策略

- **写一致性（W）**：  
  - 对每个写入/续期/撤销操作，入口节点向 N 个副本并发发送请求：  
    - `W=ONE`：任意一个副本确认即认为写入成功（最低延迟）  
    - `W=QUORUM`：收到超过半数（`⌈N/2⌉`）副本确认后认为成功（推荐默认）  
    - `W=ALL`：所有副本确认才认为成功（适合关键管理操作）  
  - 对踢人/撤销类操作，建议默认采用不少于 QUORUM 的 `W` 级别，以减小用户被踢出后仍短暂通过校验的概率。

- **读一致性（R）**：  
  - 对校验/查询操作，入口节点从副本集合中读取 `R` 份数据：  
    - `R=ONE`：读取一个副本即可（最快，可能读到旧数据）  
    - `R=QUORUM`：读取超过半数副本并合并结果（更高一致性）  
  - 配置建议：  
    - 对高 QPS 的 App 校验 API，可默认 `R=ONE`，结合后台 read-repair 在可接受时间范围内修复落后副本；  
    - 对安全敏感或管理查询场景，可以使用 `R=QUORUM`。  

- **一致性安全条件**：  
  - 在无故障情况下，满足 `R + W > N` 时可以保证读操作观察到最新写入（逻辑上）。  
  - 在有故障或网络分区时，Cluster 可以通过配置决定是偏向可用性（降低 R/W）还是偏向一致性（拒绝部分请求）。  
  - **踢人/撤销操作的特殊要求**：  
    - 鉴于撤销/踢人属于安全敏感操作，应比普通写入采用更高的一致性级别：  
      - 推荐默认至少 `W=QUORUM`，在高安全环境中可配置为 `W=ALL`；  
      - 当因网络分区或节点故障无法达到目标 W 时，应向管理端明确返回失败/部分成功信息，而不是默默降级并声称“踢人成功”。  
    - 管理 API 可提供选项允许在极端情况下“强制降级”（例如从 `W=ALL` 降为 `W=QUORUM`），但必须伴随明确告警与审计记录，以便运维方意识到一致性风险。

## 5. 节点加入（Join）与数据重平衡
 
- **步骤 1：引入新节点与 Ring 更新**  
  - 新节点加入后，集群通过成员协议将其纳入成员视图，并由 Seed Nodes 生成新的 Ring 配置（新的 `ring_epoch`）。  
  - 新 Ring 会让部分分区的副本集合包含新节点，这些分区的数据需要迁移到新节点。  

- **步骤 2：数据迁移到新节点（Rebalance In）**  
  - 对“新节点将负责的分区”，原先负责这些分区的节点会：  
    - 基于当前快照 + WAL 执行一次 bulk copy，将该分区的当前数据复制到新节点；  
    - 同时维护一个增量变更队列，将迁移过程中产生的新写入/更新/删除同步到新节点。  
  - 在迁移阶段，读写路径仍以旧 Ring 为准，保证行为可预测。  

- **步骤 3：切换生产角色**  
  - 当某分区在新节点上的数据同步完成并满足一致性要求：  
    - 更新 `ring_epoch`，正式使用新副本集合；  
    - 新写入和读取按照新 Ring 进行路由。  
  - 入口节点在进行路由计算时会携带或查询当前有效的 `ring_epoch`，避免新旧 Ring 混用。

## 6. 节点下线（Leave/Decommission）与故障

- **计划下线（Graceful Leave）**：  
  - 运维将节点标记为 `LEAVING`，集群生成新 Ring，将该节点负责的分区重新分配给其他节点。  
  - 原节点向接管节点执行数据迁移（与 Join 类似的 bulk + 增量同步流程）。  
  - 当所有目标分区的数据同步完成后，将该节点从 Ring 与成员视图中移除，并停止对外服务。

- **非计划下线（Failure/Crash）**：  
  - 通过成员协议检测到节点 `DOWN`：  
    - 若剩余副本数足以支撑当前配置的 R/W，一致性策略继续生效；  
    - 若副本不足，集群可按策略：  
      - 拒绝部分写入/查询请求（保持严格一致性）；或  
      - 允许对部分 API 降级一致性（例如校验 API 允许从 QUORUM 降到 ONE），同时记录告警。  
  - 故障节点恢复后，通过反熵机制比较数据（基于 WAL、版本号或哈希摘要）并补齐缺失部分。

- **时间与时钟同步的基本要求**：  
  - 集群在处理并发更新（续期、撤销）和故障恢复时，经常需要依赖时间戳或“最后写入”的概念来判断新旧状态。  
  - 部署环境应启用 NTP 或等效时间同步机制，并在运维规范中建议最大允许 Clock Skew（例如子秒级到若干秒级）；  
  - 在未来 SDD-1/SDD-2 设计中，应进一步明确：  
    - 对于时钟偏差超出阈值的节点，是否允许其加入 Ring；  
    - 是否在生命周期逻辑中引入小的容忍窗口（Grace Window），避免微小时间偏差导致跨节点判断不一致。

## 7. 数据平均分配与负载均衡

- **虚拟节点与均匀性**：  
  - 使用虚拟节点（vnode）均匀分布分区到物理节点，降低单次 Join/Leave 时的迁移数据量与负载波动。  

- **进一步的负载调优（可选后续工作）**：  
  - 监控各节点存储数据量、QPS、延迟等指标，在后续版本中支持：  
    - 调整 vnode 数量或权重，引导部分分区迁移到更空闲的节点；  
    - 在 Hot Key 场景下采用局部复制或特殊处理策略。  
  - v1 优先保证正确性与基本均衡，复杂策略可以在未来迭代中扩展。

## 8. 与其他设计的关系

- **与持久化设计（R11、R17，见 `TokMesh-v1-20-design-persistence.md`）**：  
  - 节点级快照与 WAL 机制为分区迁移与故障恢复提供数据基础；  
  - Join/Leave 中的数据迁移应优先利用现有持久化结构（快照 + 增量日志）实现高效复制。  

- **与安全设计（R5–R9）**：  
  - 集群内部节点间通信（包含数据迁移与一致性协议消息）应使用 mTLS 保护；  
  - Ring 与成员视图的变更也属于敏感操作，应通过管理平面与安全策略进行控制。  

- **与端口隔离与接入设计（R10、R16、R19）**：  
  - Any-Node Gateway 能力需要在业务端口与管理端口上分别落实：  
    - 业务端口主要承担高 QPS 的生命周期与校验请求；  
    - 管理端口承担集群管理与全局查询，配合适当限流。  
  - 集群内部的路由、转发与分片迁移应通过专用的集群内部通讯端口完成（受 mTLS 保护），不得复用对外业务端口，以避免安全域混淆和授权逻辑绕过。  
  - SDK 只需要感知“入口地址集合”，无需承担分片与一致性细节。

## 9. 开放问题与后续工作

- **开放问题**：  
  - 是否需要对不同租户/业务线支持不同的 `replication_factor` 或 R/W 策略？  
  - 分区迁移过程中的“时间点一致性”需要达到何种程度（强一致 vs 最终一致）？  
  - 集群规模增大时，成员协议和 Ring 更新的传播延迟上限和可接受范围是多少？  
- **后续工作**：  
  - 在 SDD-1 阶段为集群与一致性机制拆分详细子需求，并与代码包结构建立映射。  
  - 设计具体的成员协议与 Ring 元数据存储/传播机制。  
  - 编写针对 Join/Leave/Failure 场景的测试与 benchmark 计划，验证一致性与可用性边界。
