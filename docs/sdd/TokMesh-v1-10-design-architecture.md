> **基本信息**  
> - **阶段**：SDD-1（规格/冻结）草案

## 1. 目标与范围

- **目标**：在 `TokMesh-v1-01-overview.md` 和 `TokMesh-v1-02-requirements.md` 的基础上，给出 TokMesh v1 的整体架构视图，包括：  
  - 逻辑组件划分（Server、CLI、管理平面、数据平面）。  
  - 单节点架构与未来集群扩展位点。  
  - 与 SSO/IAM、App、LB/WAF 等外部系统的交互关系。  
- **范围**：  
  - 聚焦于 v1 的整体架构轮廓与阶段性演进（P1–P3），不展开具体 API/协议字段细节（协议层细节在 `TokMesh-v1-11-design-protocol.md` 中描述）。  

---

## 2. 阶段性架构视图（P1 → P3）

- **P1：单节点 / 简单拓扑 + mTLS**  
  - 形态：1 个 tokmesh-server（或少量固定节点） + tokmesh-cli + SSO/IAM 后端。  
  - 核心：会话模型、多维索引、生命周期 API、基础持久化、资源阈值、PKI/mTLS、端口隔离、基础协议与部署形态。  
- **P2：安全与加密增强**  
  - 在 P1 之上，增加内存/落盘/端到端加密、主动防御、证书轮换增强、App 直连校验、配置/文档体验、令牌类型策略。  
  - 架构上保留单节点/简单拓扑假设，不引入复杂分布式一致性。  
- **P3：集群与一致性拓展**  
  - 引入一致哈希 Ring、副本数、Any-Node Gateway、Join/Leave/Rebalance 等机制。  
  - P3 应尽量在 P1/P2 的组件边界与接口之上演进，避免推翻基础设计。  

---

## 3. 逻辑组件划分（草案）

- **tokmesh-server（会话存储节点）**  
  - 职责：会话/令牌读写、索引维护、持久化、加解密、主动防御、节点级资源管理。  
  - 接口：  
    - 业务端口（R4、R10、R13）：生命周期 API、App 校验 API、HTTP/gRPC/Redis 子集。  
    - 管理端口（R5、R6、R8、R9、R19）：节点配置、证书/密钥运维、黑白名单管理、统计查询等。  
  - 在 P3 阶段：增加内部集群通讯端口，用于节点间路由与数据迁移（与业务/管理端口隔离）。  

- **tokmesh-cli（管理 CLI）**  
  - 职责：为运维/管理员提供命令行工具，封装管理平面 API 调用。  
  - 能力：证书生成/导入、证书轮换、节点 join/leave、状态观察、黑白名单管理等。  
  - 安全：通过 mTLS 与管理端口交互。  

- **SSO/IAM 服务**  
  - 使用 tokmesh-server 的内部生命周期 API（R4），管理会话写入/校验/续期/撤销。  
  - 负责将 user_id/login_ip/device_id/tenant 等元数据写入 TokMesh。  

- **业务 App**  
  - 可选：通过 R10 的只读校验 API，直接向 TokMesh 校验 Token/Session，有助于减轻 SSO 压力。  
  - 仍然通过 SSO/IAM 完成登录流程与令牌签发。  

- **LB/WAF / 网络设施**  
  - 在 P1/P2 中承担 TLS 终止、防火墙、基础限流、防刷等职责。  
  - TokMesh 侧做基础限流与黑白名单，详细分工在安全设计与协议文档中进一步明确。  

---

## 4. 安全域与端口拓扑（与 R5/R6/R19 对齐）

- **业务数据平面端口**  
  - 对象：SSO/IAM 服务、业务 App。  
  - 协议：HTTP/gRPC、Redis 子集（迁移阶段）、TLS/mTLS。  
  - 约束：不暴露管理操作，按 R19 要求与管理端口隔离。  

- **管理控制平面端口**  
  - 对象：tokmesh-cli、IAM 管理后台。  
  - 协议：HTTP/gRPC（管理 API）、mTLS 强制、强身份认证。  
  - 职责：节点/集群管理、证书与密钥运维、安全策略配置、统计与诊断。  

- **集群内部通讯端口（P3 引入）**  
  - 对象：tokmesh-server 节点之间。  
  - 协议：基于 mTLS 的内部 RPC，承载分布式路由、心跳、分片迁移等。  
  - 要求：必须与外部业务/管理端口逻辑隔离，避免安全域混淆。

## 5. 风险点对齐与未来细化方向（占位）

> 本节用于承接 Backlog 中的关键风险，说明将如何在架构层面吸收与约束。  

- **全链路加密 vs 性能**（关联 R7/R17，INBOX-CRYPTO-PERF-001）：  
  - 在架构层面约定“字段级敏感度划分”与加密策略分级，具体算法与基准由持久化设计文档细化。  
- **跨节点踢人语义**（关联 R2、R20、R21，INBOX-KICK-CLUSTER-002）：  
  - 在集群架构中明确区分按 SessionID 踢人与按 UserID 批量踢人的复杂度与一致性成本。  
- **集群时钟与时间语义**（INBOX-TIME-CLOCK-003）：  
  - 明确对 NTP 的依赖与最大允许 Clock Skew，并在生命周期语义中引入时间容忍窗口。  
- 其余风险点将在后续迭代中补充。
