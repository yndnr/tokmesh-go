# TokMesh v1 持久化与加密设计草案（SDD-0 草案）

> **基本信息**：本文件在 R11（会话持久化与快速恢复）、R17（会话数据端到端加密存储）与 R24（数据保留与清理策略）等需求基础上，给出 TokMesh v1 的持久化与加密整体思路，用于后续 SDD-1/SDD-2 细化与实现对齐。  
> **范围**：仅讨论会话/令牌数据在内存、落盘与恢复路径上的结构与策略，不涉及具体存储引擎实现细节。  
> **关联 R**：R7、R11、R17、R24（必要时参考 R3/R12/R22）

---

## 1. 背景与问题

- **背景**：TokMesh 需要在高并发场景下提供低延迟的会话/令牌读写能力，同时要在节点重启、进程异常时尽可能避免大规模重新登录。  
- **核心问题**：  
  - 如何在**保持“内存查询为主”的高性能模型**前提下，实现安全可靠的持久化？  
  - 如何让**内存加密（R7）与落盘加密（R17）**在一个统一的威胁模型下协同工作，而不会极大拖累吞吐和恢复时间？  
  - 如何在持久化设计中预留**R11 的 RTO/RPO 控制能力**与未来扩展空间？

---

## 2. 设计目标与非目标

- **目标**：
  - **G1：查询只走内存**——所有会话/令牌的查询与索引操作只依赖内存结构（主表 + 多维倒排索引），磁盘不参与在线查询路径。  
  - **G2：顺序写 + 可控刷盘**——持久化写入采用顺序追加和批量刷盘，尽量减少随机 IO 与加密开销。  
  - **G3：端到端加密存储**——敏感会话数据在内存与落盘路径上均加密或混淆存储，与 R7/R17 对齐。  
  - **G4：可预测恢复时间**——通过快照 + 增量日志的设计，在可配置的 RTO 范围内完成重启恢复。  
  - **G5：实现简单可审计**——避免过于复杂的自定义协议和密钥轮换逻辑，优先使用可解释、可验证的结构。
- **非目标**：
  - 不在 v1 引入“跨节点强一致分布式事务”，节点内部持久化以“单节点自洽 + 集群层面由复制/路由策略保证”方式处理。  
  - 不尝试通过磁盘结构直接提供复杂的离线查询（统计分析、报表），这些需求由后续专门方案承担。

---

## 3. 内存模型：权威数据与多维索引

- **权威性**：  
  - 单个 TokMesh 节点的**内存状态**是该节点对会话/令牌的权威来源。  
  - 当节点在线时，所有读写操作以内存为准，磁盘只视为恢复介质。
- **结构建议（概念级）**：  
  - **主表（Session Store）**：以 SessionID / TokenID 为主键，存储会话核心信息（TTL、状态、敏感字段等）。  
  - **多维倒排索引**：  
    - 按 UserID、LoginIP、DeviceID、Tenant 等维度建立索引，以支持 R2 中的 O(1) 踢人与多维查询。  
    - 索引结构仅存储必要引用（指向主表），避免数据重复。  
  - **元信息**：  
    - 最近访问时间、创建时间、更新计数等，用于冷/热区分与内存淘汰策略（与 R12 对接）。

---

## 4. 落盘结构：快照 + 增量日志（WAL）

### 4.1 设计思路

- **快照（Snapshot）**：  
  - 定期将当前内存中“主表 + 关键索引”的状态序列化为一个完整快照文件。  
  - 快照写入采用“写新文件 + 原子 rename”的方式，避免产生半写文件。  
  - 快照中的敏感字段采用加密存储（与 R17 一致）。
- **增量日志（WAL / ChangeLog）**：  
  - 对会话写入/续期/撤销等操作生成紧凑的变更记录，按顺序追加写入 WAL 文件。  
  - 从最新快照时间点之后的所有变更由 WAL 负责记录。  
  - WAL 中仅包含重建所需的最小字段，并对敏感部分进行字段级加密。

### 4.2 写入路径（简化逻辑）

- **写入/续期/撤销操作**：  
  - 步骤顺序：  
    1. 在内存主表与索引结构中完成更新（保证查询语义）。  
    2. 将“变更事件”推入内存队列，由后台持久化协程批量处理：  
       - 序列化为 WAL 条目；  
       - 对敏感字段加密；  
       - 顺序写入 WAL 缓冲区，并按条数或时间阈值刷盘。  
  - 前台请求感知到的是“内存更新成功”，WAL 刷盘稍有延迟但在可控窗口内完成。

### 4.3 快照策略

- **触发条件**（可配置，示例）：  
  - 自上次快照以来 WAL 大小超过某阈值；  
  - 或时间间隔达到 N 分钟；  
  - 或运维/管理 API 手动触发。  
- **生成流程**：  
  - 创建新快照文件 `snapshot-N.tmp`，扫描当前内存中的主表与关键索引：  
    - 对会话条目进行序列化；  
    - 对敏感字段加密；  
    - 以顺序方式写入快照文件。  
  - 完成后将文件原子重命名为 `snapshot-N.dat` 并记录快照元信息（时间戳、包含的会话数量、版本号等）。  
  - 清理部分旧快照与对应的 WAL 段，避免磁盘膨胀。

## 5. 加密策略：统一的威胁模型

- **威胁模型（与 R7/R17 一致）**：  
  - 假设攻击者可能获取：  
    - 进程内存 dump；  
    - 节点磁盘内容（快照、WAL 等文件）；  
  - 目标：使攻击者在缺失密钥的前提下难以直接恢复明文令牌和敏感身份信息。
- **字段与密钥管理原则**：  
  - 将会话数据按敏感度分级：  
    - 高敏感：访问令牌、刷新令牌、用户唯一标识等 → 内存与磁盘都需要加密。  
    - 中敏感：IP、设备指纹等 → 可以考虑哈希化或加密，兼顾索引需求。  
    - 低敏感或统计型：匿名计数、聚合指标等 → 可明文存储。  
  - **Envelope Encryption**（推荐原则）：  
    - 使用数据密钥（Data Key, DK）对会话敏感字段加密；  
    - DK 本身由主密钥（Master Key, MK）加密并存放，MK 不以明文落盘，可由环境变量、系统密钥环、外部 KMS 提供。  
- **性能平衡**：  
  - 对快照/WAL 加密采用批处理、并行加密策略：  
    - 对多个记录或块一起加密，以减少每次调用的固定成本；  
    - 可以通过配置控制并行度和批大小。  
  - 在性能基准中显式评估：  
    - 加密前后写入 QPS、P99 延迟差异；  
    - 启动恢复时间的变化。

---

## 6. 启动恢复与渐进加载

- **基本恢复流程**：  
  1. 加载最新快照文件：  
     - 校验文件完整性与版本；  
     - 使用正确的密钥解密并重建内存主表与索引。  
  2. 根据快照元信息定位对应 WAL 段，自快照点之后顺序重放 WAL：  
     - 解密 WAL 条目中的敏感字段；  
     - 按时间顺序将变更应用到内存结构。  
- **渐进恢复策略（可选增强）**：  
  - 若数据量过大，可以：  
    - 优先加载“最近活跃会话”（例如通过快照中的元信息标记）；  
    - 其他冷数据由后台线程按批载入或在首次访问时按需重放。  
  - 对外表现：节点在“部分恢复完成”时即可提供服务，但需在文档中明确此阶段的行为边界。

---

## 7. 内存阈值与磁盘协同（与 R12 对齐）

- **内存阈值触发行为**：  
  - 当内存使用接近配置阈值时，可选策略包括：  
    - 优先淘汰“已包含在快照 + 已持久 WAL”中的冷会话，仅从内存删除，磁盘仍保持一份可恢复副本；  
    - 进入“部分只读模式”，限制新会话写入或控制 QPS；  
    - 向运维/监控系统发出告警。  
- **淘汰与恢复关系**：  
  - 被淘汰的冷会话，如后续再次需要（例如被访问），可以：  
    - 由后台按需加载（从快照 + WAL 重放单条或小批记录）；  
    - 或简单视为“过期/不可用”，由上层 SSO 触发重新登录（视业务策略而定）。  
  - 具体行为选择需在后续版本中结合业务需求细化。

---

## 8. Benchmark 与验证建议

- **性能基准维度**（示例）：  
  - 写入 QPS（无持久化 vs 启用 WAL vs 启用 WAL+快照+加密）；  
  - 写入 P95/P99 延迟；  
  - 启动恢复时间（不同数据量级、不同快照/WAL 组合下）。  
- **安全与一致性验证**：  
  - 对快照与 WAL 文件进行“密文检查”，确保敏感字段不存在明文泄露。  
  - 构造节点崩溃 / 中途断电场景，验证：  
    - 快照文件不会处于不可解析的半写状态；  
    - WAL 在部分写入时不会破坏整体恢复流程；  
    - 恢复后的会话状态与设计期预期一致（允许可控的 RPO）。

## 9. 开放问题与后续工作

- **开放问题**：  
  - 快照粒度是否需要支持“按分区/Shard 分片快照”，以降低单文件大小和恢复时间？  
  - 是否需要支持“只快照主表、索引按规则重建”的模式，以减小快照体积？  
  - 与未来跨节点复制机制（集群复制、再均衡）的耦合方式如何设计，避免重复持久化逻辑？
- **后续工作（面向 SDD-1/SDD-2）**：  
  - 为持久化模块拆分具体 RQ 子项，并与实现包结构建立映射。  
  - 明确具体序列化格式（例如自定义二进制格式 vs Protobuf）与版本升级策略。  
  - 设计密钥管理与轮换方案文档，说明如何与现有的证书/密钥轮换机制（R9）形成统一故事。  
  - 结合实际 prototype 的 benchmark 数据，调整本草案中的策略与参数建议。
