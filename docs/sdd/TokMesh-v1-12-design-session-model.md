## 1. 目标与非目标

- **目标**：  
  - 定义 TokMesh v1 的核心数据模型：Session 与 Token 的结构、关系与索引维度。  
  - 为生命周期 API（R4/R10）、多维索引（R2）、令牌类型矩阵（R18）、持久化与加密（R7/R11/R17）提供统一的结构基础。  
- **非目标**：  
  - 不在本文件中定义具体的序列化格式（JSON/Proto）或数据库物理布局。  
  - 不处理跨集群复制、归档/审计存储等“二级存储”场景。

---

## 2. 核心概念回顾（Session vs Token）

- **Session（会话）**：  
  - 表示“某个用户在某个客户端/设备上，通过某个认证流程完成登录”这一段持续存在的服务器端状态。  
  - 是生命周期与多维索引的“主语”：挂载用户/设备/IP/租户等上下文信息。  

- **Token（令牌）**：  
  - 是客户端或下游服务持有的访问凭证，用于代表其所属 Session 向其他系统证明身份。  
  - 一个 Session 下可以存在多个 Token，用于不同用途（访问、刷新、一致性操作、高权限管理等）。  

- **令牌类型（token_type）**（最小集）：  
  - `access`：短期访问令牌，调用频率高，是主要校验对象。  
  - `refresh`：长期刷新令牌，调用频率低，一旦泄露危害大。  
  - `admin`：高权限管理会话令牌，用于管理/API 调用，需更严格审计。  
  - 其他类型（device-bound、action、m2m）可在 P2 之后补充，但 v1 模型应为其预留扩展点。

## 3. Session 数据结构（逻辑模型）

> 以下为逻辑字段，仅用于说明语义。具体实现可按包结构拆分为内部结构体。

- **核心标识字段**：  
  - `session_id`：  
    - 类型：全局唯一 ID（建议为无含义随机 ID，或基于安全哈希）。  
    - 用途：Session 的主键，所有 Token 与索引均指向此 ID。  
  - `user_id`：  
    - 类型：字符串或整数，由上游 SSO/IAM 统一约定。  
    - 用途：支撑按用户维度索引与踢人（R2）。  

- **上下文字段**：  
  - `tenant_id`：租户或组织标识，可选。  
  - `device_id`：设备标识（如设备指纹、客户端 ID）。  
  - `login_ip`：登录时客户端 IP（由 SSO 提供）。  
  - `user_agent`：可选，客户端 UA 信息，用于风控与审计。  

- **生命周期字段**：  
  - `created_at`：会话创建时间（TokMesh 接收写入的时间）。  
  - `last_active_at`：最近一次被访问/续期/验证成功的时间。  
  - `expires_at`：会话过期时间（绝对时间），由 SSO/IAM 提供或根据策略计算。  
  - `status`：会话状态，枚举：`active` / `revoked` / `expired` / `locked` 等。  

- **安全 / 风控字段**：  
  - `risk_flags`：标记风险状态（例如地理异常、多次失败）。  
  - `meta`：小规模的键值对，用于附加业务元数据（例如登录渠道、MFA 状态），不参与主索引。

## 4. Token 数据结构（逻辑模型）

- **核心标识字段**：  
  - `token_id`：  
    - 类型：令牌本身的标识，可为外部 Token 的哈希或内部生成 ID。  
    - 注意：建议在存储层只保存 Token 哈希或加密后的表示，避免直接存储明文 Token。  
  - `session_id`：关联的 Session 主键。  

- **类型与生命周期字段**：  
  - `token_type`：令牌类型（如 `access` / `refresh` / `admin` / ...）。  
  - `issued_at`：令牌签发时间。  
  - `expires_at`：令牌过期时间。  
  - `status`：令牌状态，枚举：`active` / `revoked` / `expired`。  

- **安全字段**：  
  - `scopes` / `permissions`：可选，描述令牌的权限范围。  
  - `binding`：可选，用于表达与设备/客户端/网络环境的绑定关系。  

- **注意**：  
  - 对于访问令牌，TokMesh 可以只存储验证所需的最小信息（如哈希、关联 Session、TTL、状态），Token Payload 可由 SSO/IAM 单独解析。  
  - 对于刷新令牌和高权限令牌，应视为高敏感字段，在 R7/R17 的加密与持久化策略中赋予最高保护级别。

## 5. 索引设计与查询场景（R2）

- **主索引**：  
  - `session_id` → Session 记录（及其下 Token 集合）。  
  - Token 层面通过 `session_id + token_id` 组合进行定位。  

- **二级索引（倒排索引）**（实现时需可配置开启）：  
  - `user_id` → [session_id...]  
  - `tenant_id` → [session_id...]  
  - `device_id` → [session_id...]  
  - `user_id + device_id` / `user_id + tenant_id` 等组合索引（按实际需求与成本评估选择）。  

- **典型查询场景**：  
  - 按 SessionID 查询：O(1)，用于鉴权与单会话踢人。  
  - 按 UserID 查询：O(1) 级别命中倒排索引，再遍历用户所有 Session；在集群情况下，可能需要跨节点聚合（对应 INBOX-KICK-CLUSTER-002 的权衡）。  
  - 按 DeviceID/租户 查询：用于设备级踢人、租户下线等管理操作。  

- **约束与建议**：  
  - 索引数量与组合需谨慎控制，避免写入路径过重；可按配置打开/关闭某些维度。  
  - 对于“按用户批量踢人”在集群模式下，要在集群设计中明确其复杂度与一致性语义（R20/R21）。

## 6. 生命周期挂钩与状态转换（R3/R4）

- **创建（CreateSession）**：  
  - 写入 Session 主记录与必要索引；  
  - 在需要时同步创建初始 Token（如 access + refresh）。  

- **校验（ValidateToken / ValidateSession）**：  
  - 通过 `token_id` 或 `session_id` 定位，检查状态与 TTL，并可选择更新 `last_active_at`（需考虑频率与写放大）。  

- **续期（ExtendSession / ExtendToken）**：  
  - 更新 `expires_at` 与必要索引（如过期队列），保证原子性与并发安全。  

- **撤销 / 踢人（RevokeSession / RevokeToken）**：  
  - 将 Session 或指定 Token 状态置为 `revoked`，并在索引上及时反映。  
  - 对按用户/设备维度的批量撤销，需要结合索引与集群设计定义清晰的行为。

## 7. 与加密、持久化、集群的关系（简要）

- **加密（R7/R17）**：  
  - 对高敏感字段（Token 哈希、刷新令牌、高权限 Token、UserID 等），需在内存与落盘路径上遵循统一加密策略。  
  - 对非敏感元数据（TTL、LastActive、计数类字段），可优先保持明文，以降低加解密开销并简化索引。  

- **持久化（R11/R17）**：  
  - Session 与 Token 结构需适配快照 + WAL 模式，根据字段敏感度决定落盘加密粒度。  
  - 在 Key Versioning 策略下，结构中应预留“版本/epoch”字段，便于关联所用数据密钥版本（参见 INBOX-KEY-ROTATION-006）。  

- **集群（R20/R21）**：  
  - SessionID 是分片的自然候选键；  
  - UserID/DeviceID 等索引在集群环境中可能需要跨节点扫描或全局索引才能支持“按用户踢人”的近实时语义。
