# TokMesh v1 概览（Overview）

> **目标**：用一份精简文档描述 TokMesh v1 的整体定位、问题背景、核心目标与能力维度，作为后续所有需求和设计的上位参考。

## 1. TokMesh 是什么？

- **定位**：  
  TokMesh 是一套面向身份认证（IAM/SSO）场景的 **专用 Session/Token 存储与管理系统**，目标是替代传统方案（如 Redis）在 SSO 中的会话存储角色，而不是通用 KV 数据库。

- **核心特征**：  
  - 专注于 Token/Session 的存储与校验，而非任意键值数据。  
  - 原生支持多维会话索引（User / Client / App / Tenant 等）。  
  - 内置会话全生命周期管理（创建 / 校验 / 续期 / 撤销 / TTL 清理）。  
  - 内建安全与零信任能力（PKI/mTLS、内存加密、防重放、黑白名单等）。  
  - 支持分布式部署与快速恢复（持久化 + 内存阈值管理）。  

## 2. TokMesh 解决什么问题？

- **Redis 在会话场景中的局限**：  
  - 数据模型偏通用 KV，缺乏原生多维会话索引。  
  - TTL/过期清理机制对 SSO 场景不够友好，容易产生抖动与热点。  
  - 安全相关能力（PKI、mTLS、防重放等）通常依赖外部组件或自研。  
  - 会话写入、校验、续期、撤销逻辑大量堆叠在业务代码上，演进成本高。

- **TokMesh 的目标**：  
  - 专门针对 **身份认证流量**（登录 / 校验 / 续期 / 踢人）优化数据结构与协议。  
  - 将会话相关的性能、可靠性与安全能力下沉到基础设施层，而不是留在业务代码里拼接。

## 3. 核心角色与视角

- **R1. TokMesh 开发者（TokMesh Developer）**  
  - 关注点：TokMesh 内部实现（存储引擎、多维索引、安全、防御、协议栈、部署工具）。  
  - 职责：  
    - 实现和演进 TokMesh Server / CLI / SDK。  
    - 确保性能、稳定性与安全红线达标。  
  - 常用内容：内部设计文档、协议与数据模型说明、测试与 benchmark 结果。  

- **R2. SSO 开发/管理者（SSO Developer/Admin）**  
  - 关注点：现有或新 SSO 系统如何使用 TokMesh 存储和校验会话/令牌。  
  - 职责：  
    - 旧 SSO：从 Redis 迁移到 TokMesh（使用 Redis 子集或 SDK）。  
    - 新 SSO：直接使用 TokMesh 的生命周期 API 和 SDK。  
    - 在写入 TokMesh 时提供必要元数据（如 user_id / login_ip / device 等），以支持多维索引与踢人。  
  - 常用内容：接入指南、生命周期 API 文档、SDK 文档、迁移指南。  

- **R3. TokMesh Server 管理员（TokMesh Server Admin）**  
  - 关注点：TokMesh 实例本身的部署、运行与基础运维。  
  - 职责：  
    - 在不同环境（Windows/Linux/容器/K8S）中安装、启动和升级 TokMesh Server/CLI。  
    - 处理节点级别的监控、日志、磁盘与内存、水位报警等基础设施问题。  
    - 按照 IAM 的安全策略要求，落地具体配置（证书文件、端口、防火墙等）。  
  - 常用内容：部署文档、配置示例、运维手册、告警与排障指南。  

- **R4. IAM 开发/管理者（IAM Developer/Admin）**  
  - 关注点：身份与安全策略整体，TokMesh 是其中一个重要组件。  
  - 职责：  
    - 决定 TokMesh 的部署拓扑与安全策略（PKI、mTLS、防御规则）。  
    - 通过管理 API/CLI 做证书轮换、黑白名单管理、内存阈值配置等。  
    - 监控 TokMesh 状态，参与容量规划和性能调优。  
  - 常用内容：安全与运维指南、管理 API / CLI 文档、部署策略与配置规范。  

- **R5. 业务系统开发/管理者（Application Developer/Admin）**  
  - 关注点：业务系统如何消费 SSO 与 TokMesh 提供的会话/令牌能力。  
  - 职责：  
    - 决定是否在部分场景下直接调用 TokMesh 的只读校验 API（R10），减轻 SSO 压力。  
    - 在客户端/服务端集成 TokMesh SDK 或 REST 调用，正确处理校验结果。  
    - 与 SSO/IAM 协同处理踢人、异常登录等业务逻辑。  
  - 常用内容：R10 相关文档、SDK 接入指南与示例代码。  

- **R6. 终端用户（End User）**  
  - 关注点：登录是否稳定、响应是否迅速、被踢/过期等行为是否符合预期。  
  - 不直接接触 TokMesh 文档或 API，其体验通过 SSO/业务系统的行为间接体现。

## 4. 能力维度（后续需求的分组依据）

TokMesh v1 的需求可以大致分成以下几个能力维度：

- **功能维度**：  
  - 会话数据模型与生命周期语义。  
  - 会话生命周期 API（创建 / 校验 / 续期 / 撤销）。  
  - 多维会话索引与踢人能力。  
  - App 直连的只读校验 API。

- **安全维度**：  
  - 零依赖外部 PKI 的证书管理（基于 Go 标准库）。  
  - 全链路 mTLS 与零信任网络假设。  
  - 内存数据加密。  
  - 主动防御（重放防御、自动黑名单、白名单、豁免 IP）。  
  - 证书与通讯密钥的集群级轮换。

- **可靠性与资源维度**：  
  - 会话数据持久化与快速恢复。  
  - 内存阈值与节点资源管理。

- **部署与体验维度**：  
  - 打包与部署形态（Server/CLI、Windows/Linux/容器/K8S）。  
  - 配置体验与示例文档（配置文件结构、注释、样例）。  

这些维度在 `TokMesh-v1-02-requirements.md` 中会被具体展开为一条条需求（R1, R2, ...），并在后续设计文档中对应到实现与测试细节。

## 5. 会话（Session）与令牌（Token）模型

- **会话（Session）**：  
  - 表示“某个用户在某个客户端/设备上，通过某个认证流程完成登录”这一段持续存在的服务器端状态。  
  - 会话是生命周期的“主语”，承载用户标识、设备信息、登录 IP、租户、创建时间、最后活动时间以及风控相关标记等。  
  - 多维索引（User / IP / Device / Tenant 等）主要作用在会话层，用于支持 R2 中的快速查询与踢人。  

- **令牌（Token）**：  
  - 是客户端或下游服务持有的“访问凭证”，用于代表其所属会话向其他系统证明身份。  
  - 一个会话下可以存在多个不同用途与类型的令牌（例如短期访问令牌、刷新令牌、一次性操作令牌等），对应 R18 中的令牌类型矩阵。  
  - 令牌可以被轮换、撤销，而不一定立即终止整个会话（具体行为由 SSO/IAM 策略决定）。  

- **TokMesh 内部关系（概念表示）**：  
  - 会话是主对象：`Session{ session_id, user_id, device_id, login_ip, tenant, ... }`。  
  - 会话下挂载一组令牌：`Token{ token_id, token_type, ttl, 状态, 绑定信息... }`。  
  - 踢人 / 撤销行为可以针对：  
    - 某个会话及其所有令牌（按用户/设备/租户维度批量操作）；  
    - 会话下的某类令牌（例如只失效高权限管理令牌）。  
  - 存储与协议设计应以“会话为核心，令牌为附属凭证”的思路展开，避免将二者混为单一扁平对象。

## 6. API 分类与平面

- **业务数据平面（Business APIs）**：  
  - 面向 SSO/IAM 服务与业务 App。  
  - 承载会话生命周期操作（创建 / 校验 / 续期 / 撤销）以及只读校验接口。  
  - 典型需求：R2、R3、R4、R10、R13。

- **管理控制平面（Management/Admin APIs）**：  
  - 面向 TokMesh 管理员 / IAM 运维 / `tokmesh-cli`。  
  - 承载节点管理、配置变更、证书与密钥轮换、安全策略管理等操作。  
  - 典型需求：R5、R6、R8、R9、部分 R14、R15。

- **可观测性平面（Observability APIs）**：  
  - 面向监控与审计系统。  
  - 提供指标、健康检查与审计日志输出等能力。  
  - 典型需求：将来在监控/审计相关 R 条目中完善（当前仅在性能与测试红线中有全局约束）。

## 7. 质量红线（全局）

- **性能优先**：  
  - 针对会话核心操作（创建 / 校验 / 续期 / 踢人），必须有明确的性能指标与 benchmark 计划。  

- **测试覆盖率目标 > 80%**：  
  - TokMesh 核心模块和对外 API 应通过单元测试和集成测试达到至少 80% 的覆盖率。
