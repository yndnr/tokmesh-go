# TokMesh v1 术语表（Glossary）

> **目标**：统一 TokMesh v1 文档与讨论中的关键术语含义，减少歧义，方便人类与 AI 在需求、设计和实现阶段保持一致理解。

---

## 1. 基本概念

| **术语**      | 定义 |
|:--------------|:-----|
| **Session（会话）** | 表示“某个用户在某个客户端/设备上，通过某个认证流程完成登录”这一段持续存在的服务器端状态。Session 是生命周期与多维索引的核心对象，承载用户标识、设备信息、登录 IP、租户、创建时间、最后活动时间等信息。 |
| **Token（令牌）** | 客户端或下游服务持有的访问凭证，用于代表其所属 Session 向其他系统证明身份。一个 Session 下可以有多个 Token，不同 Token 可用于不同用途（访问、刷新、一次性操作等）。 |
| **Token Type（令牌类型）** | 用于区分不同用途与风险级别的令牌类别，例如：短期访问令牌、长期刷新令牌、设备绑定令牌、一次性操作令牌、高权限管理令牌、服务间机器令牌等。不同类型对应不同的 TTL、索引与审计策略（参见 R18）。 |
| **TTL（Time To Live）** | 会话或令牌的生存时间配置，到期后系统可以自动将其视为失效并执行清理。TTL 既适用于 Session，也适用于 Token。 |
| **踢人（Kick / Revoke）** | 管理侧对某个用户/设备/会话执行的主动撤销行为，可终止一个或多个 Session 及其下的 Token，用于风控、下线、权限回收等场景。 |

## 2. 系统与架构相关

| **术语**      | 定义 |
|:--------------|:-----|
| **TokMesh Server** | TokMesh 的服务端进程，负责会话/令牌的存储、查询、生命周期管理以及对外暴露各类协议接口（Redis 子集、HTTP、gRPC 等）。 |
| **TokMesh CLI（tokmesh-cli）** | 面向运维与管理员的命令行工具，用于管理 TokMesh Server 与集群：节点管理、配置下发、证书与密钥运维、调试和诊断等。 |
| **业务数据平面（Business Data Plane / Business APIs）** | 承载会话生命周期与令牌校验相关业务数据操作的 API 平面，包括 SSO/IAM 内部使用的创建/校验/续期/撤销接口，以及对 App 暴露的只读校验接口（R4、R10 对应）。 |
| **管理控制平面（Management Control Plane / Admin APIs）** | 承载节点管理、配置变更、证书与密钥轮换、安全策略与黑白名单管理等操作的 API 平面，主要由 TokMesh 管理员和 IAM 管理员通过 `tokmesh-cli` 或集成系统使用（对应 R5、R6、R8、R9、R14、R15 的部分内容）。 |
| **可观测性平面（Observability Plane）** | 用于监控与审计的接口集合，包括指标导出、健康检查、审计日志输出等能力，为后续监控/审计相关需求预留空间。 |

## 3. 安全与证书相关

| **术语**      | 定义 |
|:--------------|:-----|
| **PKI（Public Key Infrastructure）** | 公钥基础设施，这里指 TokMesh 依赖的证书与密钥体系。TokMesh 要求尽量使用 Go 标准库实现 PKI 能力，不强制依赖 OpenSSL 等外部库（参见 R5）。 |
| **mTLS（Mutual TLS）** | 双向 TLS 认证，客户端与服务端互相验证对方证书身份。TokMesh 在集群内部通信、管理控制平面以及 SSO/IAM 服务接入场景下建议或要求使用 mTLS（参见 R6、R9）。 |
| **零信任（Zero Trust）** | 一种安全假设：即便是在“内网”，网络环境也不被默认信任，所有访问都需要经过严格的身份认证、加密与授权控制。TokMesh 将自身设计在零信任前提下。 |
| **内存数据加密** | 对 TokMesh 进程内存中存放的敏感 Session/Token 数据进行加密或混淆，减少进程内存被 dump 或系统被入侵时的泄露风险（参见 R7）。 |
| **落盘加密** | 对快照、WAL 等持久化文件中的敏感数据进行加密存储，使攻击者在仅获得磁盘内容而无法访问密钥时难以恢复明文数据（参见 R17）。 |

## 4. 角色相关

| **术语**      | 定义 |
|:--------------|:-----|
| **TokMesh 开发者（TokMesh Developer）** | 负责 TokMesh Server/CLI/SDK 的设计与实现，关注存储引擎、多维索引、安全能力、协议栈和性能（对应概览文档中的 R1 角色）。 |
| **SSO 开发/管理者（SSO Developer/Admin）** | 负责现有或新 SSO 系统的开发与运维，将 TokMesh 集成为会话/令牌存储后端，负责在写入 TokMesh 时提供必要元数据（user_id、login_ip、device 等）。 |
| **TokMesh Server 管理员（TokMesh Server Admin）** | 负责 TokMesh 实例的部署、运行与日常运维管理，包括节点生命周期、资源监控与基础配置。 |
| **IAM 开发/管理者（IAM Developer/Admin）** | 负责整体身份与安全策略，将 TokMesh 纳入 IAM 体系，通过管理平面进行证书、策略与风控配置。 |
| **业务系统开发/管理者（Application Developer/Admin）** | 负责业务系统如何与 SSO/TokMesh 集成，是否以及如何使用 TokMesh 的只读校验 API 减轻 SSO 压力。 |
| **终端用户（End User）** | 通过 SSO 与业务系统间接感知 TokMesh 能力的最终用户，不直接接触 TokMesh 接口与文档。 |

## 5. 使用建议

- 后续在需求、设计与实现文档中出现新术语时，建议：  
  - 先在文档内首次出现处用括号解释；  
  - 再根据需要将其补充到本术语表中，保持跨文档的一致含义。  
- 当 AI 协助撰写文档时，应优先使用本术语表中的术语及定义，避免同一概念使用多个不同名称导致的混淆。
