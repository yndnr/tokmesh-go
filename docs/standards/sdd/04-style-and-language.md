# TokMesh-Go SDD 写作与排版规范

> 目标：让 SDD 文档在保持信息密度的前提下，层次清晰、美观统一，便于人类与 AI 阅读和生成。

本规范适用于 `docs/sdd/` 下的所有 SDD 文档，以及 `docs/standards/sdd/` 下的规范文档示例。

---

## 1. 标题与层次

- **单一主标题**：每个文件只使用一个 `#` 级标题，作为文档名称。  
- **主体层级**：主体章节使用 `##`，小节使用 `###`，一般不超过三级。  
- **编号习惯**：建议在章节标题中加入编号，例如：  
  - `## 1. 基本信息`  
  - `### 3.1 RQ-TM-XXXX-A：子需求标题`
- **章节顺序**：推荐的通用顺序为：  
  - **基本信息 → 背景与问题 → 目标与非目标 → 场景 → 方案 → 约束 → 结论 / 验收**

---

## 2. 粗体与信息槽位

- **关键槽位用粗体**：在段落或列表中，用粗体标记信息名称，例如：  

  ```markdown
  ## 1. 基本信息

  - **RQ-ID**：RQ-TM-XXXX  
  - **标题**：Mesh Join 机制优化  
  - **状态**：draft-0
  ```

- **块级摘要**：需要快速综述时，可使用粗体块名作为视觉锚点，例如：

  ```markdown
  - **基本信息**：……  
  - **背景与问题**：……  
  - **目标与非目标**：……  
  - **场景**：……  
  - **方案**：……  
  - **约束**：……  
  - **结论**：……
  ```

- **节标题不再额外加粗**：标题本身通过 `# / ## / ###` 表达层次，不重复使用粗体，以保持简洁。

---

## 3. 列表、表格与代码块

- **列表用途**：  
  - 用短句列表承载要点，避免长段落。  
  - 一条列表项只表达一个核心信息。

- **表格用途与对齐规则**：  
  - 结构化信息（如 RQ-ID 映射、测试矩阵、benchmark 一览）统一使用 Markdown 表格。  
  - 表头使用简短名词，必要时配合正文解释。  
  - **文本列**：列中主要为文本内容时，使用左对齐（`:---` 或 `:---:`，推荐 `:---`），例如：

    ```markdown
    | **子需求 ID** | 主要实现位置                        | 备注 |
    |:-------------|:------------------------------------|:-----|
    | RQ-TM-XXXX-A | internal/xxx/xxx.go:FuncA           | …    |
    | RQ-TM-XXXX-B | internal/yyy/yyy.go:TypeB / MethodB | …    |
    ```

  - **数值列**：列中主要为数字或数值型指标时，使用右对齐（`---:`）：

    ```markdown
    | 指标              | 目标值 |
    |:------------------|------:|
    | P95 延迟 (ms)     |   100 |
    | 吞吐 (QPS)        |  2000 |
    ```

  - **重要文本列加粗**：当某一列承载“数据行标题”或特别重要的文本（如 RQ-ID、用例名、指标名）时：  
    - 表头文本使用粗体（如 `**子需求 ID**`）；  
    - 如有需要，可对行内该列内容使用粗体以突出（如 `**RQ-TM-XXXX-A**`）。

- **代码与命令块**：  
  - 命令、路径、标识符一律使用反引号：`go test ./... -cover`、`internal/mesh/`。  
  - 跨行命令或示例使用 fenced code block，并在前后各留一行空行：

  ```markdown
  ```bash
  go test ./... -cover
  ```
  ```

---

## 4. 文本风格与语言

- **语言**：  
  - 文档主体以中文为主；必要技术术语可保留英文（如 Mesh、benchmark）。  
  - 英文术语首次出现时可在括号中给出英文原文或缩写。  
- **语气**：  
  - 使用客观、指令式语气，避免口语化表达。  
  - 示例、建议可用“建议 / 可选”等字样明确区分强制与非强制内容。
- **段落长度**：  
  - 避免超过 5 行的长段落，必要时拆分为多个小段或列表。  
  - 每段聚焦一个主题。

## 5. 常见结构推荐

### 5.1 SDD-0（探索）推荐结构

- **1. 基本信息**  
- **2. 背景与问题描述**  
- **3. 目标与非目标**  
- **4. 关键使用场景（Use Cases）**  
- **5. 候选方案**  
- **6. TokMesh 特有技术约束初步识别**  
- **7. 初步结论与开放问题**

### 5.2 SDD-1（规格）推荐结构

- **1. 基本信息**  
- **2. 范围与上下文**  
- **3. 需求列表（按 RQ-ID 子项）**  
- **4. 非功能需求（NFR）与质量属性**  
- **5. 验收标准总览**

### 5.3 SDD-2（实现与测试）推荐结构

- **1. 基本信息**  
- **2. 模块与包结构设计**  
- **3. 接口与数据模型**  
- **4. RQ-ID → 实现映射**  
- **5. 错误处理与边界条件**  
- **6. 性能与资源考虑（与 benchmark 关联）**  
- **7. 部署与运维要点**  
- **8. 测试策略与用例矩阵**  
- **9. 覆盖率目标与 benchmark 设计**  
- **10. 验证与结论**

---

## 6. AI 写作与审查要求

当 AI 参与撰写或修改 SDD 时，应：

- **遵守本规范**：标题层级、粗体槽位、表格和代码块使用方式需与本文件一致。  
- **保持结构稳定**：除非有充分理由，不随意增删推荐章节，而是在推荐章节内部补充内容。  
- **显式标注信息槽位**：重要字段（如 RQ-ID、状态、性能目标、覆盖率目标等）必须使用粗体标签标出。  
- **进行排版自检**：在输出前简单检查：标题层级是否连续、表格是否闭合、代码块是否成对。

如发现已有文档与本规范不一致，AI 可在保持语义不变的前提下，对排版进行最小必要的修正，以提高整体可读性和一致性。
