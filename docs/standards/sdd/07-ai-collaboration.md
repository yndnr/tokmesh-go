# TokMesh-Go AI 协作与提示词规范

> **目标**：约束人类与 AI 在 TokMesh-Go 项目中的协作方式，让所有提示词（prompts）都围绕 SDD-0 / SDD-1 / SDD-2 流程、有组织地推进需求、设计、实现、重构与审核。  
> **适用对象**：所有参与本项目的 AI（Gemini / Claude / Codex / GPT 等）以及与 AI 交互的人类开发者。

本规范与 `00-constitution.md` 配套使用：  
- 宪法负责定义 **必须遵守的规则**；  
- 本文件负责给出 **如何提问 / 如何驱动 AI 工作** 的提示词模板。

---

## 1. 提示词使用的基本原则

- **显式声明阶段与目标**：  
  在每次对话开头注明当前所处阶段（SDD-0 / SDD-1 / SDD-2）和本轮目标（如“澄清需求 / 起草规格 / 设计实现 / 设计测试 / 重构 / 审核”等）。

- **指定 AI 角色**：  
  明确要求 AI 以 **虚拟架构师** 或 **合规审查员**（或两者兼具）的角色工作，例如：  
  - “你现在以虚拟架构师的身份，帮我……”  
  - “你现在以合规审查员的身份，检查……”

- **绑定文档与 RQ-ID**：  
  尽可能在提示词中指出：  
  - 相关 RQ-ID（如 `RQ-TM-0001`）。  
  - 对应文档路径（如 `docs/sdd/RQ-TM-0001-mesh-join/00-SDD-0-exploration.md`）。

- **结构化表达需求**：  
  使用分点、分小节方式描述问题和期望输出，避免让 AI 猜测意图。

- **要求给出合规结论**：  
  在关键步骤后，提示 AI 按宪法要求给出“合规 / 部分合规 / 不合规”的结论与下一步建议。

## 2. SDD-0（探索）阶段提示词模板

> **目标**：澄清问题空间与方案空间，不进入实现细节，为 SDD-1 提供材料。

### 2.1 需求澄清 / 讨论模板

**场景**：已有一个模糊想法，需要 AI 帮忙拆解成背景、目标、场景、候选方案。

```text
【阶段】SDD-0 探索
【角色】虚拟架构师
【RQ-ID】暂定为 RQ-TM-XXXX（尚未在 registry.md 登记）

【任务】
帮我把下面这段想法整理成 SDD-0 探索文档的骨架，不要写代码：
- 背景与问题
- 目标与非目标
- 关键使用场景（至少 3 个）
- 候选方案（至少 2 个）
- TokMesh 特有约束的初步识别
- 初步结论与开放问题

【原始想法】
...

请输出为一份结构化的大纲，符合 docs/standards/sdd/04-style-and-language.md 的排版风格。
```

### 2.2 从讨论转化为 SDD-0 文档模板

**场景**：已经讨论过一轮，想让 AI 直接生成或更新 `00-SDD-0-exploration.md` 内容。

```text
【阶段】SDD-0 探索
【角色】虚拟架构师
【RQ-ID】RQ-TM-XXXX（目录：docs/sdd/RQ-TM-XXXX-xxx/）

【任务】
根据我们前面讨论的内容，帮我生成/更新
docs/sdd/RQ-TM-XXXX-xxx/00-SDD-0-exploration.md 的正文：
- 使用模板 docs/sdd/_template-RQ-TM-XXXX-example/00-SDD-0-exploration.md 的章节结构。
- 填入我们讨论过的背景、目标、场景、候选方案与初步结论。
- 对不确定的地方，用“开放问题”列出来，而不是编造细节。

【已有讨论摘要】
...
```

## 3. SDD-1（规格 / 冻结）阶段提示词模板

> **目标**：从 SDD-0 提炼出可验证的需求规格（RQ-ID 列表、约束和验收标准）。

### 3.1 从 SDD-0 提炼规格

```text
【阶段】SDD-1 规格 / 冻结
【角色】虚拟架构师
【RQ-ID】RQ-TM-XXXX
【文档】
- SDD-0：docs/sdd/RQ-TM-XXXX-xxx/00-SDD-0-exploration.md

【任务】
请基于上述 SDD-0：
1. 为该需求定义或拆分出若干子 RQ-ID，如 RQ-TM-XXXX-A/B/...
2. 为每个子 RQ-ID 补全以下内容：
   - 描述
   - 触发条件
   - 期望行为
   - 错误与异常场景
   - TokMesh 技术约束绑定
   - 验收标准（包含测试视角）
3. 输出为一份符合模板 docs/sdd/_template-RQ-TM-XXXX-example/01-SDD-1-specification.md 结构的文稿。

【注意】
- 不要设计具体代码或接口签名，只聚焦“应该做什么”。
```

### 3.2 规格评审 / 补全提示词

```text
【阶段】SDD-1 规格 / 冻结
【角色】虚拟架构师 + 合规审查员
【RQ-ID】RQ-TM-XXXX
【文档】
- SDD-1：docs/sdd/RQ-TM-XXXX-xxx/01-SDD-1-specification.md

【任务】
1. 请审阅上述 SDD-1 文档，从以下角度指出缺口：
   - RQ-ID 是否足够原子且不重叠？
   - 错误与异常场景是否覆盖关键情况？
   - TokMesh 特有技术约束是否被明确绑定？
   - 验收标准是否可操作、可测试？
2. 对每一处缺口，给出具体修改建议，必要时给出示例文本。
3. 最后按 00-constitution.md 的格式给出合规结论：
   - SDD 流程
   - TokMesh 技术约束
   - 质量红线（这里只需要规划层面）
```

## 4. SDD-2（实现 / 交付）阶段提示词模板

> **目标**：将规格映射到实现设计、测试和 benchmark 规划，指导后续编码与重构。

### 4.1 实现设计（模块 / 接口 / 数据）提示词

```text
【阶段】SDD-2 实现设计
【角色】虚拟架构师
【RQ-ID】RQ-TM-XXXX
【文档】
- SDD-1：docs/sdd/RQ-TM-XXXX-xxx/01-SDD-1-specification.md

【任务】
请基于上述 SDD-1，起草 SDD-2 实现设计：
1. 设计涉及的 Go 包和模块结构（cmd/internal/pkg），说明各自职责。
2. 为关键接口给出函数签名草案（包名 + 函数名 + 参数 + 返回值）及行为说明。
3. 定义主要数据结构（结构体字段及含义），以及与 TokMesh 协议 / 消息的映射。
4. 建立 RQ-ID → 实现映射表。

【输出格式】
- 遵循 docs/sdd/_template-RQ-TM-XXXX-example/02-SDD-2-implementation.md 的章节结构。
- 不写完整代码，只到可以进入编码的“接口与数据模型”层面。
```

### 4.2 测试与 Benchmark 设计提示词

```text
【阶段】SDD-2 测试与 Benchmark 设计
【角色】虚拟架构师 + 合规审查员
【RQ-ID】RQ-TM-XXXX
【文档】
- SDD-2 实现设计：docs/sdd/RQ-TM-XXXX-xxx/02-SDD-2-implementation.md

【任务】
基于 SDD-2 实现设计，为该 RQ 制定测试与 benchmark 方案：
1. 设计单元测试、集成测试、端到端测试的大致范围。
2. 按 RQ-ID 子项列出测试用例矩阵（正向 / 异常 / 边界）。
3. 设定覆盖率目标（默认 >= 80%），指出关键包的覆盖要求。
4. 规划需要的 benchmark：
   - 哪些路径需要 benchmark？
   - 对应的 benchmark 函数名称、所在包与文件？
   - 主要观测指标（延迟、吞吐、CPU、内存等）。
5. 最终输出为 docs/sdd/_template-RQ-TM-XXXX-example/03-SDD-2-tests-benchmarks.md 样式的文稿。
```

## 5. 代码相关场景的提示词规范

> **前提**：只有当对应 RQ 的 SDD-2 已存在并绑定到 SDD-1 / RQ-ID 时，才允许进入“正式编码”提示。

### 5.1 编写或修改代码

```text
【阶段】代码实现（基于 SDD-2）
【角色】执行者 + 合规审查员
【RQ-ID】RQ-TM-XXXX
【文档】
- SDD-2 实现设计：docs/sdd/RQ-TM-XXXX-xxx/02-SDD-2-implementation.md
- 测试设计：docs/sdd/RQ-TM-XXXX-xxx/03-SDD-2-tests-benchmarks.md

【任务】
根据上述 SDD-2 设计：
1. 在指定文件/包中实现对应的函数或结构体。
2. 为每个实现新增或补充单元测试，朝覆盖率 >= 80% 的目标前进。
3. 如涉及关键路径，实现相应的 benchmark 函数。

【约束】
- 不得偏离 SDD-2 中已有的设计；如发现设计有问题，请先建议更新 SDD-2。
- 实现完成后，请给出一次合规审查（特别关注测试与 benchmark）。
```

### 5.2 代码重构提示词

```text
【阶段】代码重构（可能影响 SDD-2）
【角色】虚拟架构师 + 执行者 + 合规审查员
【RQ-ID】RQ-TM-XXXX
【文档】
- SDD-2 实现设计：...

【任务】
我想对以下代码进行重构，以改进可读性/性能/抽象结构：
（附上相关代码片段或文件路径）

请你：
1. 先从架构和 SDD-2 的角度评估重构是否合理，有无违反既有设计边界。
2. 提出不改变外部行为的重构方案（说明重构目标与步骤）。
3. 指出重构可能影响的测试与 benchmark，提醒需要补充或更新的测试。
4. 实施重构时，确保与 SDD-2 中的 RQ-ID → 实现映射保持一致或及时更新。
```

## 6. 审核 / 合规检查提示词规范

> **目标**：让 AI 以合规审查员身份，系统性地审查文档与代码是否符合宪法与 SDD 流程。

### 6.1 文档合规检查

```text
【阶段】文档审核
【角色】合规审查员
【对象】
- SDD-0：...
- SDD-1：...
- SDD-2：...

【任务】
请从以下三个维度对上述文档进行合规审查：
1. SDD 流程合规性：
   - SDD-0 → SDD-1 → SDD-2 的链路是否清晰、完整？
   - 是否出现跳过某一阶段直接进入后续阶段的情况？
2. TokMesh 技术约束：
   - 是否违反已有约束（语言版本、协议、拓扑、安全等）？
   - 是否存在需要在约束文档中补充的新假设？
3. 质量红线（规划层面）：
   - 性能目标与 benchmark 规划是否合理？
   - 测试覆盖率目标是否明确且可达？

最后按 00-constitution.md 建议的格式，给出“合规结论 / 问题列表 / 建议的下一步”。
```

### 6.2 代码与测试合规检查

```text
【阶段】代码与测试审核
【角色】合规审查员
【RQ-ID】RQ-TM-XXXX
【文档与代码】
- 相关 SDD 文档：...
- 相关代码文件：...

【任务】
请检查：
1. 代码是否严格围绕 SDD-2 中定义的接口、数据结构与行为实现？
2. 是否存在未在 SDD-2 中记录的“隐式需求”或“额外行为”？
3. 测试是否覆盖所有 RQ-ID 子项的正向 / 异常 / 边界场景？
4. 是否已实现必要的 benchmark（如已规划）？

【输出】
- 给出合规结论（合规 / 部分合规 / 不合规）。
- 按问题列出需补充的文档、测试、benchmark 或重构建议。
```

## 7. 执行与任务拆分提示词规范

> **目标**：让 AI 帮助拆解任务、制定执行计划，并在每步结束时进行轻量审查。

### 7.1 任务拆分与执行计划

```text
【阶段】任务规划
【角色】虚拟架构师 + 执行者
【范围】
- 目标：完成 RQ-TM-XXXX 的 SDD-0 → SDD-2 → 初版实现与测试。

【任务】
请帮我：
1. 拆解为若干可执行的子任务（按 SDD-0 / SDD-1 / SDD-2 / 代码 / 测试 / benchmark 分组）。
2. 为每个子任务给出简短描述和预期产出物（文档路径或代码路径）。
3. 标记哪些步骤必须在“文档到位”后才能进行。
4. 用简洁的列表形式给出执行顺序。
```

### 7.2 阶段性回顾提示词

```text
【阶段】阶段性回顾
【角色】合规审查员
【任务】
基于我们刚刚完成的工作，请总结：
1. 本轮完成了哪些 SDD 或代码层面的变更（按文件列出即可）。
2. 从 SDD 流程、TokMesh 约束、质量红线三方面评估合规情况。
3. 列出下一阶段的 3～5 个优先任务。
```

## 8. 使用建议小结

- **在每条提示词中显式标明：阶段 / 角色 / RQ-ID / 文档路径 / 任务**。  
- **优先驱动 AI 完成 SDD-0 / SDD-1 / SDD-2 文档，再进入代码与测试层面**。  
- **在关键阶段结束时，主动请求“合规审查”输出结构化结论**。  
- **当发现需求变更或设计演进时，先更新相关 SDD，再让 AI 修改代码**。

遵循以上提示词规范，可以让 Gemini / Claude / Codex / GPT 等不同 AI 在同一套「语言与流程」下协作，持续围绕 SDD-0 → SDD-1 → SDD-2 晋级制度与 TokMesh 的技术约束推进项目。
