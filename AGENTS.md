# Repository Guidelines

本文件面向本仓库的所有贡献者（包括 AI 代理）。除非子目录存在更具体的 `AGENTS.md`，否则本文件适用于当前目录下的全部代码与文档。

## 项目结构与模块组织

- 入口程序：`cmd/tokmesh-server/`（`main.go` 与 CLI 相关代码），保持逻辑尽量精简。
- 核心业务与基础设施：`internal/`（如 `api/`、`config/`、`server/`、`session/`）为主要实现位置。
- 共享或实验性代码：`src/` 用于通用组件或实验模块。
- 规范与文档：`docs/` 和 `docs/standards/sdd/`，其中包含 SDD 与写作规范，修改前**必须**先阅读对应规范。

## 文档与 SDD 规范

- `docs/` 与 `docs/standards/sdd/` 下的所有 SDD 文档，必须遵循 `docs/standards/sdd/00-constitution.md` 中的通用要求。
- 写作与排版需严格按 `docs/standards/sdd/04-style-and-language.md` 执行：标题层级、粗体槽位、表格、代码块等需保持一致。
- AI 或人类在修改 SDD 时，不应随意增删推荐章节；如需调整结构，应在 SDD 中明确说明原因。

## SDD 流程与质量红线

- 严格执行“文档优先 (Doc-First)”：没有在 SDD 中登记和说明的需求/改动，不应直接进入代码实现。
- 新功能、重大改动和关键 bug 修复，应沿 SDD-0 → SDD-1 → SDD-2 流程演进，并在 `docs/sdd/` 中可追溯到对应的 R 编号或 RQ-ID。
- 代码合并前，相关实现应能从 SDD-2 映射回已冻结的规格（SDD-1），并在测试设计和验收标准中体现。
- 遵守质量红线：性能优先，核心路径需要有明确的测试与 benchmark 计划；整体测试覆盖率目标为 ≥ 80%，关键包不低于该标准。

## 构建、测试与开发命令

- `go build ./...`：检查所有包是否能成功编译。
- `go build ./cmd/tokmesh-server`：构建主服务二进制。
- `go test ./... -cover`：运行全部测试并统计覆盖率。
- 开发时默认在仓库根目录执行上述命令，确保 `go.mod` 模块路径正确解析。

## 代码风格与命名约定

- 使用 Go 官方推荐风格，函数保持短小、职责单一。
- 提交前运行 `gofmt`（或 `go fmt ./...`），确保导入与缩进规范。
- 导出类型和函数使用 `CamelCase`，局部变量使用 `camelCase`；仅在文件名或配置键需要时使用 `snake_case`。
- `internal/` 中对外可复用的接口应补充 GoDoc 注释，并尽量保持兼容性。

## 测试规范

- 测试文件与被测代码同目录，命名为 `xxx_test.go`。
- 优先采用表驱动测试，覆盖正常路径、异常输入与边界条件（包括并发场景）。
- 保持 `go test ./... -cover` 通过，整体覆盖率目标 ≥ 80%；修复缺陷时优先增加回归测试，并在需要时同步更新 SDD 中的测试/验收条目。
 - 开发节奏建议采用“小步实现 + 立即测试”的方式：每完成一个小能力块（如一个包或一组接口）的核心逻辑后，应尽快补充并运行相应的单元测试或最小集成测试，而不是在大范围改动完成后一次性集中补测。

## 提交与 Pull Request 规范

- 单次提交聚焦一个变更点，提交信息建议采用 `模块: 简要说明`，例如：`server: fix session timeout`。
- 在提交或 PR 描述中关联相关 Issue（如 `Fixes #123`），并简要说明动机与影响。
- PR 描述应包含：变更概览、影响范围（特别是协议或运维层面）以及本地运行过的测试命令。

## AI 代理补充说明

- 修改任意文件前，先检查目标目录是否存在更细粒度的 `AGENTS.md`，并遵循其中更严格的规则。
- 当修改 `docs/` 下 SDD 文档时，必须先阅读并严格遵守相关 SDD 规范文件，如发现排版不符合规范，可在不改变语义的前提下顺手修正。
- 优先进行局部、小步修改；如需大规模重构或结构调整，应在单独 PR 中说明理由与方案。
- 处理需求、设计或重构相关请求时，优先以“虚拟架构师 + 合规审查员”角色工作：先补齐 SDD-0/1/2，再给出正式实现方案；对绕过文档阶段的请求，应明确标记为“非合规建议（仅供探索）”。
